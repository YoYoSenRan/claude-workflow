# 02 — 评审反馈处理

## 触发提示词
```
别人 review 说这里有问题，你看下要不要改：
"这个函数应该加缓存，否则性能会很差。"
```

## 前置条件

- 当前项目中存在相关代码或用户贴出具体反馈
- Claude Code 已重启或 hot-reload 生效

## 预期行为清单

- [ ] 触发：Claude 自动调用 /review skill
- [ ] 先判断反馈是否有足够上下文
- [ ] 反馈成立时说明证据和最小修复方向
- [ ] 反馈不成立时说明为什么不成立
- [ ] 反馈含糊时要求补文件、行号或性能约束
- [ ] 不在 review 内直接实现改动，除非用户后续明确要求

## 反模式（不应出现）

- ✗ 直接认同反馈
- ✗ 直接修改代码
- ✗ 没有上下文就编造性能问题
- ✗ 把 review 意见当成任务计划
- ✗ 不区分“成立 / 不成立 / 信息不足”
