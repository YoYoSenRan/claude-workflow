如果你是被派来执行特定任务的子代理, 跳过此 skill。任务已在主智能体拆好, 不要再走一遍 Load/Review/TodoWrite 循环。

# 执行实现计划

加载 `plan` 产出的计划文件, 评审 → TodoWrite → 逐任务执行 → 阻塞点就停。

<HARD-GATE>
遇到阻塞点 (测试失败 / 依赖缺失 / 指令不清 / 实现跟计划不符) → **停**, 报告用户。

禁止: 跳步骤 / 自己改计划 / 猜测意图 / "差不多就行"。

计划是契约, 不是建议。
</HARD-GATE>

## 反模式: "我看一眼计划就开干"

最常掉的坑。直接跳到 Task 1 = 漏掉关键评审 = 带 bug 上路。

计划写错的概率 > 0。先评审再勾清单。

## 清单

按顺序完成:

1. **Load** — 读计划全文 (Read 工具, 一次性)
2. **关键评审** — 找阻塞点 (歧义 / 缺依赖 / 顺序倒挂)
3. **TodoWrite 建** — 每任务 = 一项
4. **逐任务执行**:
   - 标为 in_progress
   - **判子代理派遣**（见下文"子代理决策"段）
   - 主智能体自做 / 派遣子代理执行 / 并行派遣多子代理
   - 按步骤走, 验证 expected
   - 标为 completed
5. **遇阻塞点停** — 不猜不试, 报告用户
6. **完工** — 全部 done → 调用增强 skill (verification-before-completion → finishing-branch) 或交付

## 流程图

```
Load 计划
   ↓
关键评审 ── 有阻塞点 ──→ 报告用户, 暂停
   ↓ 通过
TodoWrite 建项
   ↓
取下一任务
   ↓
标为 in_progress
   ↓
按步骤执行 + 验证
   ↓
通过? ── 否 ──→ 阻塞点? ── 是 → 停, 问
   ↓ 是                     └ 否 → 修, 重试 (最多 1 轮)
标为 completed
   ↓
还有任务? ── 是 ──→ 取下一
   ↓ 否
提示后续 (finishing)
```

## 关键评审

加载计划后, 动手前, 静默问自己:

- [ ] 任务顺序无依赖倒挂?
- [ ] 文件路径都存在? (Create 例外)
- [ ] 命令能跑? (依赖装了吗?)
- [ ] 步骤之间无遗漏?
- [ ] 我能理解每步骤干啥?

任一答"否" → 阻塞点, 报告用户, 不动手。

## 阻塞点类型 + 处理

| 阻塞点 | 处理 |
|---|---|
| 测试失败超 1 轮重试 | 停, 报告: "任务 N 步骤 M 测试持续失败, 输出: ..." |
| 文件 / 命令未找到 | 停, 报告: "任务 N 假设 X 存在, 实际不存在" |
| 步骤含糊不能落地 | 停, 报告: "任务 N 步骤 M 含糊, 候选解读: A / B" |
| 实现跟规格不符 | 停, 报告: "任务 N 实现 X, 但计划写的是 Y" |
| 修改超出计划范围 | 停, 报告: "任务 N 需改文件 Z, 但计划没列" |

**不要做的**:

- ✗ 跳过失败步骤继续下一任务
- ✗ 自己改计划
- ✗ 猜测用户意图
- ✗ 编 expected 输出

## TodoWrite 用法

加载计划后立刻建项:

```
TodoWrite([
  { content: "Task 1: <name>", status: "pending" },
  { content: "Task 2: <name>", status: "pending" },
  ...
])
```

每任务开始 → 标为 `in_progress`; 完成 → 标为 `completed`。

步骤级别**不**进 TodoWrite (粒度太细, 用计划自己的清单)。

## TDD 处理

计划怎么写, 你怎么执行。

- 计划第 1 步骤是"写失败测试" → 老实写, 跑, 看到红
- 计划没测试步骤 → 不要自己加 (违反计划即契约)

要问的: 计划该不该有测试? → 这是计划阶段的事, 不归你。

## 子代理决策

每个任务进 in_progress 时, 按下表决定执行方式:

| 任务特征 | 执行方式 |
|---|---|
| 单文件 / 单步骤 / 简单替换 / 配置改动 | 主智能体自做 |
| 跨多文件 / 需独立验证 / 上下文重 | 派遣子代理 (每任务一个新 agent) |
| 多个互不依赖任务 (任务 N + N+1 没有依赖) | **并行**派遣多个子代理 (单条消息内多 Agent 调用) |
| 含 review / audit / 安全审计性质 | 派遣 reviewer 类子代理 (caveman:cavecrew-reviewer / security 等) |
| bug 排查类子任务 | 派遣 debugger 子代理 |
| 纯查找定位 / 不需修改 | 派遣 Explore / cavecrew-investigator |

**派遣示例** (Agent 工具):

```
单任务:
  Agent({ description, subagent_type, prompt })

并行多任务 (单条消息, 多 Agent 调用):
  Agent({ ... task A ... })
  Agent({ ... task B ... })
  Agent({ ... task C ... })
```

**派遣前必做**:

- prompt 写完整 (子代理不见对话历史): 目标 / 输入 / 期望产出 / 验证方式
- 标明产出形式 (报告 / diff / 文件列表)
- 子代理不递归走主流程 (SUBAGENT-STOP 已阻断)

**子代理回报后**:

- 主智能体接回报, 标 task completed
- 子代理结果不可见用户 → 主智能体复述关键产出
- 子代理失败 → 按"阻塞点处理"段走

## 前后衔接

- **前序**: 计划产出的 `docs/plans/*.md`
- **后续**: 全部任务完成 → 调用增强 skill (按路线图顺序):
  - `verification-before-completion` (尚未实现 → 暂自行跑测试 + 自审)
  - `finishing-branch` (尚未实现 → 暂自行 commit + PR)
  - 增强 skill 实现前, 收尾保持轻量, 由你自行做

## 警示信号

| 内心戏 | 真相 |
|---|---|
| "步骤大概是这意思, 直接干" | 大概 ≠ 确定。停, 问 |
| "测试失败可能环境问题, 跳过" | 跳过 = 隐藏 bug |
| "这步骤没写代码, 我补一下" | 补 = 改计划。停, 问 |
| "计划漏了 X, 我顺手加" | 顺手 = 范围扩散。停, 问 |
| "我先把简单任务全做了" | 顺序破坏 = 依赖倒挂风险 |
| "阻塞点等会再处理" | 现在就停, 继续 = 累积错误 |
| "TodoWrite 太重, 不建了" | 建。进度可追踪 = 用户能审 |

## 核心原则

- **HARD-GATE 不可破** — 阻塞点必停
- **计划即契约** — 不擅自改
- **不猜不试** — 模糊就问
- **TodoWrite 必建** — 进度可追踪
- **逐项验证** — 每步骤 expected 必跑
