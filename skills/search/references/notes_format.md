# 子代理笔记格式合同

子代理 P2 阶段交付物必须遵守的格式。主代理依赖此格式做 P3 合并。

## 三段结构

### ① Sources — 来源段

```
[X1] {作者/机构} — {标题} | URL | type: {标签} | date: YYYY-MM-DD | auth: 1-10
```

字段含义 —

| 字段 | 必填 | 说明 |
|---|---|---|
| `X` | ✓ | 任务字母 (A/B/C...), 配合编号唯一 |
| 作者/机构 | ✓ | 原始出处, 不是聚合站 |
| 标题 | ✓ | 实际页面标题 |
| URL | ✓ | 真访问过的完整链接, 不准缩短/编造 |
| `type` | ✓ | 见下表 |
| `date` | ✓ | 文章发布日期 (找不到精确日, 至少给年月) |
| `auth` | ✓ | Authority 1-10, 见评分卡 |

**type 取值**:

| 标签 | 含义 | 例子 |
|---|---|---|
| `official` | 第一手官方 | 公司 blog、SEC 文件、政府报告、GitHub 官方仓库 |
| `academic` | 同行评议 | 期刊论文、会议论文、博士论文 |
| `secondary-industry` | 专业分析 | 行业研报、分析师覆盖、专业媒体深度文 |
| `journalism` | 新闻报道 | 主流媒体、调查报道 |
| `community` | 用户生成 | 论坛、评论、社交媒体、Q&A 站 |
| `other` | 混合/未明 | 聚合站、未核实源 |

**Authority 1-10 评分卡**:

| 分 | 标准 |
|---|---|
| 9-10 | 一手官方 + 强信誉 (公司官方 + 政府/学术) |
| 7-8 | 知名媒体 / 行业头部分析师 / 高 star GitHub 维护中项目 |
| 5-6 | 一般行业博客、中型媒体、活跃个人专家 |
| 3-4 | 营销页、聚合站、内容农场 |
| 1-2 | 匿名贴、内容明显有偏 |

### ② Findings — 发现段

```
- {单句事实} [X1][X3]
- {单句事实} [X2]
```

规则 —
- 每条**单句**, 不要"另外/此外"扩展
- 每条**至少 1 个 `[Xn]`**, 多源支撑就多列
- 数字 + 日期 + 单位齐全
- 不确定标 `[unverified]` 或不写
- ≤15 条, 多了说明粒度太细
- 优先放可量化、可对比的事实, 而非感受性描述

### ③ Gaps — 缺口段

至少**一条具体内容**, 覆盖以下任一类:

```
- 搜了但没找到: {具体什么问题没答案}
- 反向解释候选: {如果当前结论错, 可能错在哪}
- 时效风险: {哪些源已 >18 月}
```

**Gaps 不是缺点 — 是诚实的边界声明**。允许写"无明显 gap"但必须紧接说明"已交叉验证 N 条主张, 未发现对立证据"等具体依据 — **不接受单字"无"**。

## 可选段: Deep Reads

仅 DEEP 任务才有。2-3 篇全文蒸馏:

```
### [X1] {标题}
- **关键数据**: {3-5 条带数字的事实}
- **核心观点**: {作者主张}
- **反例/限制**: {哪些场景作者承认不适用}
```

## 主代理校验清单

收到子代理笔记后, 主代理过一遍 (per-task 阈值, 与 SKILL.md P2 同步) —

- [ ] Sources 段: Standard 4-8 条 / Lightweight 3-5 条
- [ ] 每条 Source 字段齐全 (URL/type/date/auth)
- [ ] Findings 每条都有 `[Xn]` 源号
- [ ] type 不全是 `community` 或 `other`
- [ ] 至少有一条 official 或 academic
- [ ] 有 Gaps 段, 且含至少一条具体内容 (非单字"无")

不达标 — 把笔记发回, 让子代理补 (或主代理自己补一轮搜索)。

注: 全局 (注册表) 阈值在 SKILL.md P3 Quality Gates。本表只管单任务。
