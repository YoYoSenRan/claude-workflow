# 最终报告骨架

P4 起草用本模板。可按需增删章节, 但**摘要 / 正文 / 反向争议 / 引文注册表** 四块必备。

## 顶部元信息

```
# {报告标题}

时效基准: YYYY-MM-DD | 模式: 常规/深度 | 视角: {调研立场}
来源: N 条 (M 域名, 官方+学术 K%)
```

## 摘要

3-5 句, 直接给结论。每个核心主张带 `[n]` 引用。

```
## 摘要

{结论 1, 一句话, 带源} [3][7]

{结论 2} [1][12]

{结论 3} [5]
```

## 正文章节

每节遵循 — 论点 → 证据 → 信心度 → 反向声明 (如有)。

```
## N. {章节标题}

{论点段落, 每条事实带 [n]}。

| 关键数据 | 数值 | 来源 |
|---|---|---|
| ... | ... | [n] |

**信心度**: 高/中/低 — {一句理由, 如"3 个独立官方源" / "仅 1 个厂商营销页"}

> 反向声明: 亦有 {另一观点}, 见 [n]。
```

**章节排序原则** — 主题先于任务序。读者不该感知"这是任务 A 写的、那是任务 B 写的"。

## 反向争议段 (P5 自审"类型 B"问题才进这里)

```
## 核心争议

**争议 1**: {主张 A [n] vs 反向证据 B [m]} — {为何重要 + 倾向判断}
**争议 2**: ...
```

只放有**实质对立证据**的项 — 缺源、时效问题、内部矛盾属于 P5 悄悄修类型 (类型 A/C), 不写进这里。所以 0-N 条都正常, **不强求凑 3 条**。零争议时写: "本次未发现实质对立证据的争议项 — 自审已覆盖 X/Y/Z 角度, 未发现反向声明"。

## 信心度概览

```
## 信心度概览

| 章节 | 信心度 | 主要约束 |
|---|---|---|
| 1. {标题} | 高 | 多源官方 |
| 2. {标题} | 中 | 单源 + 厂商背景 |
| 3. {标题} | 低 | 二手数据, 待核 |
```

## 引文注册表 (附录)

放报告末尾。**P3 锁源后的"通过"列表**, 全文用 `[n]` 引用此处。

```
## 引文注册表

[1] {作者/机构} — {标题}
    URL: ...
    类型: 官方 | 日期: YYYY-MM-DD | 权威: 9 | 状态: ✓ 畅通
[2] {作者/机构} — {标题}
    URL: ...
    类型: 社区 | 日期: YYYY-MM-DD | 权威: 5 | 状态: ⚠ 代理 (r.jina.ai)
[3] {作者/机构} — {标题}
    URL: ...
    类型: 行业分析 | 日期: YYYY-MM-DD | 权威: 6 | 状态: ⚠ 快照 (2025-03-08)
...

(可选) 淘汰:
× {来源} — 原因: {单源 + 权威<5 / 时效过期 / 与 [N] 重复 / ✗ 访问受限}

统计: 通过={n}/{total}, 域名={k}, 官方占比={pct}%, ⚠ 转手源占比={p}%
```

(统计格式与 SKILL.md P3 注册表段一致, 必要时统一更新两处)

## 局限声明段

```
## 局限

- {缺什么数据没找到}
- {时效基准日之后可能过时的部分}
- {方法论限制 — 如只查了英文源、没访问付费数据库}
- 访问受限源: {标 ✗ 受限的站点列表 + 影响范围, 如"LinkedIn 公司页 / Crunchbase 融资数据无法核, 涉及章节 2 / 3 部分声明"}
- ⚠ 转手源占比: {p}% — 注册表中代理/快照拿到的源占比, 偏高 (>40%) 时整体信心度下调
```

诚实声明边界 = 报告可信度的一部分。引文里有 ✗ 受限源时**必须**在本段明说哪些章节受影响。

## (技术调研专项段 — 触发时必填)

### 决策矩阵 (放在摘要后, 正文章节前)

```
## 决策矩阵

| 维度 | {候选 A} | {候选 B} | {候选 C} |
|---|---|---|---|
| 维护活跃度 (最后提交) | 2026-04 ✅ | 2024-11 ⚠️ | 2026-05 ✅ |
| 星标 / 生态 | 45k | 12k | 88k |
| 许可证 | MIT ✅ | AGPL-3.0 ⚠️ | Apache 2.0 ✅ |
| 包体积 / 安装大小 | 12KB | 45KB | 8KB |
| 学习曲线 | 低 | 中 | 高 |
| 团队栈契合 | {依据本地代码库} | ... | ... |
| 性能 (基准测试) | {数字 + 源} | ... | ... |
| 趋势位置 | Adopt ✅ | Hold ⚠️ | Trial 🟡 |
| 主要风险 | 单点维护者 | 许可证病毒式 | 学习成本 |

**推荐**: {候选} — {一句话理由 + [n] 引用}
```

未做技术调研时本段可省。维度按候选特性增删, 但 最后提交 / 许可证 / 趋势位置 / 团队栈契合 四行必填。"趋势位置"取自 Tech Radar 圈层 (Adopt/Trial/Assess/Hold) 或 State of XX 满意度百分比 — 见 SKILL.md TR-8。

### 代码片段段 (正文中插入, 不集中堆末尾)

每个候选在正文章节里至少 1 段官方代码示例:

````
### {候选 A} 基础用法
```{lang}
{≤15 行调用示例, 引自子代理的代码片段段}
```
源: [n]
````

不要抄二手博客代码 — 必须可溯源到官方文档。

### 实测建议 (放在引文注册表前)

````
## 实测建议 (最小验证)

最小验证命令:
```bash
{npm/pip/cargo install ...}
{最小 hello-world 代码或 cli 调用}
```

观察点:
- {点 1, 如"启动时间"}
- {点 2, 如"内存占用"}
- {点 3, 如"错误信息可读性"}

预计耗时: {N 分钟}
````

文献综述不等于决策 — 本段告诉用户怎么花 30 分钟亲手验证。

## 不要做的事

- ✗ 用编号引用不在注册表里的源
- ✗ 章节里出现"根据搜索结果..."这种露馅句
- ✗ 抄子代理笔记原句 — 要重新组织
- ✗ 摘要写得比正文还含糊
- ✗ 每节都"信心度: 高" — 自我膨胀的信号
- ✗ 把"缺源"、"时效过期"这类 P5 类型 A 问题硬塞进"核心争议"凑数 (这些应悄悄修, 见 SKILL P5)
- ✗ (技术调研) 不报最后提交日期就推荐库
- ✗ (技术调研) 决策矩阵省略"团队栈契合"行 — 这是开发者调研的核心维度
- ✗ (技术调研) 推荐库不给一段官方代码示例

## 长度参考

| 模式 | 总字数 | 章节数 | 引用源 |
|---|---|---|---|
| 常规模式 | 2000-4000 | 3-5 | ≥6 |
| 深度模式 | 4000-7000 | 5-8 | ≥10 |

超过上限多半是子代理笔记原样搬入 — 回去蒸馏。
