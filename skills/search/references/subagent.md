# 子代理派遣

主代理拆任务后, 用本模板派遣并发子代理 (Agent tool, subagent_type=general-purpose)。

## 模板 (填占位符后整段做 prompt)

````
你是调研子代理。

## 角色
{专家角色, 如 "Vue3 admin 模板研究员" / "AI 浏览器自动化分析师"}

## 目标
{一句话调查目标}

## 时效基准
{今天日期 YYYY-MM-DD} — 超 18 个月的资料标 [未核实]

## 搜索查询 (起手 2-3 个)
1. "{query 1}"
2. "{query 2}"
3. "{query 3}"

## 抓取规约
WebFetch 失败 (HTTP 403 / 429 / Cloudflare 拦截 / 登录墙 / 返回 < 5 行) 时立即执行代理级联:

  WebFetch → curl https://r.jina.ai/{URL} → curl https://defuddle.md/{URL} → archive.org → 标失败

GitHub 走 raw.githubusercontent.com 或 `gh api`。PDF 走 r.jina.ai 或 pdftotext。
同一 URL 同方法最多重试 1 次 (共 2 次), 第 3 次起立即切换下一级联节点。

## 输出格式

# 任务 {ID}: {主题}

## 来源 (3-5 条)
[X1] {作者/机构} - {标题} - URL - 类型 ({官方/学术/行业/新闻/社区/其他}) - 日期 - 权威 1-10 - 状态 ({✓ 畅通 / ⚠ 代理 + 服务名 / ⚠ 快照 + 日期 / ✗ 受限})
[X2] ...

## 发现 (≤5 条核心点, 每条带 [Xn])

**前 2 条必须直答用户原题**, 不写背景定义。例:
- 问 "Vue3 admin 选哪个", 前 2 条应是: "vue-vben-admin 32k★, 国产第一 [X1]" 而不是 "Vue 是渐进式框架 [X1]"

**剩余 ≤ 3 条**: 直接影响选型决策的关键数据 (维护活跃 / 生态规模 / 体积 / 学习成本)。

**反模式 (禁止)**:
- **不发散周边话题** — 任务问 A, 不主动调研 B/C/D 横向选项
- **不主动横向扩展** — 任务未点名的候选, 不纳入笔记
- **不堆基础知识** — Vue / React 等技术的定义与历史, 不写

## 缺口
- 已搜索但未获取到的具体数据点
- 时效风险 (源 > 18 个月)
- 访问失败的 URL (笔记内标 ✗ 受限)

## (技术调研专项, 命中 library/framework/SDK/库对比/选型 才填)
每个候选收齐:
- stars / last_commit / latest_release / open issues / license / 维护方
- 包体积 (Bundlephobia / Packagephobia, 失败可标 [未核实])
- 1-2 段官方代码片段

## 铁律
- 不许编 URL, 只用实际访问过的链接
- 不许凭印象写数字, 找不到标 [未核实]
- 不许塞整页 HTML, 提炼成发现
- **笔记 ≤ 1200 中文字** (含发现 + 来源 + 缺口三段总和)
- **不发散** — 任务里没问的横向选项 / 背景知识 / 周边话题, 一律不写

开始。
````

## 派遣建议

- **默认 2 个子代理, 上限 3** — 超 3 须用户原题里点名 3+ 候选
- **工作集不重叠** — 2 个子代理覆盖范围互不重叠, 合并完整覆盖原题
- **来源不重叠** — 不同任务搜不同域
- **默认 SCAN** (摘要够用), 关键源才 DEEP (WebFetch 全文)
- **同一数据点不重复派子代理验证** (例: GitHub 星数已由 A 子代理拿到, 不再派 B 验证)
