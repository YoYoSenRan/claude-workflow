# 子代理派遣模板

P2 阶段, 主代理用本模板派遣并行子代理。把占位符 `{...}` 填上后整段作为子代理的 prompt。

## 模板

````
你是深度调研流水线的子代理。

## 角色
{专家角色, 如 "TLS 指纹研究员" / "AI 浏览器自动化分析师"}

## 意图标签
{主意图: P0 表中的标签, 如 "评估库选型"} {副意图: 可选}

## 首选源 (按意图查 TR-1 场景表自动填)
- 第一站: {如 "GitHub repo + context7 MCP"}
- 第二站: {如 "Bundlephobia + Packagephobia"}
- 兜底: web_search

**规则** — 先打第一站, 再第二站, 不到万不得已不直接 web_search 拿二手博客。

## 目标
{一句话调查目标}

## 时效基准
{YYYY-MM-DD} — 超过 18 个月的资料视为可能过时, 标 [未核实] 或降信心度

## 搜索查询 (先用这些, 不够再扩)
1. "{查询 1}"
2. "{查询 2}"
3. "{查询 3}"

## 深度
{深读 — 用 WebFetch 拉 2-3 篇关键全文 / 速读 — 摘要够用}

## 抓取规约 (见 references/fetch-methods.md)

WebFetch 失败 (403 / 429 / Cloudflare / 登录墙 / 返回 < 5 行) 时, 立刻按代理级联走:

  WebFetch → curl https://r.jina.ai/{URL} → curl https://defuddle.md/{URL} → archive.org → 标 ✗ 受限

GitHub 内容走 raw.githubusercontent.com 或 `gh api`, 不走代理级联。
PDF 走 r.jina.ai 或本地 pdftotext / marker_single。

**同一 URL 同一方法最多打 2 次 (1 次原始 + 1 次重试)** — 第 3 次起视为死磕, 立刻走下一站, 别烧 token。

## 输出格式 (整段返回, 别加客套话)

# 任务 {ID}: {主题}

## 来源
[X1] {作者/机构} — {标题} | URL | 类型: {官方/学术/行业分析/新闻/社区/其他} | 日期: YYYY-MM-DD | 权威: 1-10 | 状态: {✓ 畅通 / ⚠ 代理 (服务名) / ⚠ 快照 (日期) / ✗ 受限}
[X2] ...
(深度模式 4-8 条 / 常规模式 3-5 条; 按首次出现编号, X 是任务字母如 A/B/C)

## 发现 (≤15 条, 每条单句 + 源号)
- {单句事实} [X1][X3]
- ...

## 深度精读 (深读任务才有, 2-3 篇)
### [X1] {标题}
- 关键数据: ...
- 核心观点: ...
- 反例/限制: ...

## 缺口
- 搜了但没找到: ...
- 反向解释候选: ...
- 时效风险: ...
- 访问失败: {标 ✗ 受限的 URL 列表, 简述失败原因}

## 自评信心度
整体: 高/中/低 — 一句理由

## (技术调研专项 — 命中 SKILL.md TR 段触发关键词时必填)

### GitHub / 包管理元数据 (每候选一条)
{库名}:
  stars: N
  last_commit: YYYY-MM-DD       (>12 月标 ⚠️ 维护风险)
  latest_release: vX.Y.Z @ YYYY-MM-DD
  issues: {open}/{total}
  license: MIT/Apache-2.0/AGPL-3.0/...
  bundle_size_gzip: KB           (Bundlephobia)
  install_size: MB               (Packagephobia)
  maintainer: {公司/个人/基金会}

数据源选取按 SKILL.md TR-1 场景驱动表 (按调研意图选源, 不再线性优先级)。本任务的"首选源"字段已由主代理预填。

### 代码片段 (每候选 1-3 段, 整任务 ≤5 段)
### [Xn] {库名} 基础调用
```{lang}
{≤15 行核心调用示例, 从官方文档/作者代码抠}
```
铁律: 只抠官方文档/作者 README/example 的代码, 不抓 Medium/dev.to 二手教程。

## 铁律
- ✗ 不许编 URL — 只用 web_search / WebFetch 实际访问过的链接
- ✗ 不许凭印象写数字 — 找不到就标 [未核实]
- ✗ 不许把整页原始 HTML 塞回来 — 蒸馏成发现
- ✗ 不许对同一 URL 同一方法打 ≥3 次 — 失败立刻走代理级联
- ✗ 不许把 Cloudflare 拦截页 ("Just a moment...") 当真实内容存入笔记
- ✓ 每条发现必须带 [Xn] 源号
- ✓ 每条来源必须带状态字段 (✓ / ⚠ / ✗)
- ✓ 数字带日期 + 单位
- ✓ 厂商自家营销页注意打折 — 多源交叉
- ✓ 整份笔记不超过 ~2000 中文字 (~3000 英文词)

开始。
````

## 派遣建议

- **并发 ≤3** — 子代理太多易超 API 限速
- **角色分工** — 同一主题不同切面 (如调研"反爬": 一个查检测端、一个查工具端、一个查云服务)
- **来源多样** — 不同任务尽量让搜索域不重叠
- **深读慎用** — 每次 WebFetch 耗 token, 默认速读, 关键源才深读
