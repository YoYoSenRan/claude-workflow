# Search: 深度调研工作流

并行派出多个子代理搜资料 → 主代理只读蒸馏笔记 → 锁定引文后起草 → 反向自审 → 交付。

核心承诺 — **每条事实必带 `[n]` 引用, 主代理不读原始搜索结果, 引文 P3 后锁死不可加新源**。

## 适用 / 不适用

| 适用 | 不适用 |
|---|---|
| 多源对比、技术选型、竞品/行业分析、政策综述 | 单页问答 (直接 WebFetch 即可) |
| 用户要求"可追溯"、"无幻觉"、"带出处" | 已知答案的快速核对 |
| 主题跨 2 个以上子领域 | 单文件理解 (用 Read 工具) |
| 需要写成报告交付 | 纯写作任务 (本 skill 出调研报告, 不做润色) |

判断不准 — 默认常规模式 (见 P0)。用户明说"深度调研 / 综述 / 多源对比 / 技术选型 / 行业分析"或主题跨 3+ 子领域时升级到深度模式。

## 架构

```
主代理 (协调器, 上下文轻)
  │
  P0: 范围 + 时效基准设定
  │
  P1: 任务板分解 (3-5 任务)
  │
  P2: 并行派子代理 ──→ 子代理 A → 任务 A 笔记 ─┐
                    ──→ 子代理 B → 任务 B 笔记 ─┤
                    ──→ 子代理 C → 任务 C 笔记 ─┘
  │                                            │
  │     主代理只读笔记 ←───────────────────────┘
  │     (子代理的原始搜索结果留在它自己上下文, 用完即弃)
  │
  P3: 合并去重 → 引文注册表 [n] (锁源)
  │
  P4: 大纲 + 起草 (每事实带 [n] + 信心度)
  │
  P5: 反向自审 (≥3 问题) + 核验 + 交付
```

**省上下文关键** — 子代理把搜来的几十 KB 网页蒸馏成 ≤15 条发现, 主代理上下文减负 60-70%。

## 技术调研专项

**自动叠加** — 问题涉及以下任一时, 在 P0-P5 主流程基础上叠加本节规则:

触发关键词: library, framework, SDK, API, npm, pip, crate, gem, 技术选型, 库对比, 框架对比, benchmark, 性能对比, 工具对比, GitHub repo, 开源项目, 维护活跃度

### TR-1: 数据源场景驱动表 (覆盖默认)

按调研意图选源, 不再线性优先级。意图标签来自 P0 识别。

| 调研意图 | 第一站 | 第二站 | 兜底 |
|---|---|---|---|
| 查 API / 规范 | MDN / W3C / WHATWG / TC39 / framework 自家文档 | context7 MCP | DevDocs 聚合 / Read the Docs |
| 评估库选型 | GitHub repo + context7 MCP | Bundlephobia + Packagephobia + libraries.io | npm/pypi/crates.io 元数据 |
| 看技术风向 | Thoughtworks Tech Radar 最新卷 | State of JS / CSS / HTML / Devs (Devographics) | JetBrains Developer Ecosystem / GitHub Octoverse |
| 实时趋势 | Hacker News / Lobsters | InfoQ / Twitter dev 圈 | web_search |
| 问新问题 | GitHub Discussions / 各 framework Discord | 限定 subreddit (r/webdev / r/ExperiencedDevs 等) | (不推 Stack Overflow 提新问题) |
| 搜旧答案 | Stack Overflow (注意 2024-25 提问量降 64%) | GitHub Issues 历史 | web_search |
| 中文圈 | 掘金精选 (juejin.cn) / InfoQ 中文 | 各 framework 中文官方文档 | (慎用 CSDN / 思否 / V2EX — 信号噪声比低) |

**铁律** —
- 不要直接跳过第一站用 web_search 拿二手博客
- 中文调研不要拿英文社区数据强解释中国情况 — 走中文圈一行
- 一个任务可叠加 2 行 (主意图 + 副意图), 两套源都打过

### TR-2: 子代理强制采集字段

每个候选 (库/框架/工具) 至少收齐:

| 字段 | 标签 | 时效红线 | 来源建议 |
|---|---|---|---|
| GitHub 星标 | `stars: N` | — | GitHub repo |
| 最后提交日期 | `last_commit: YYYY-MM-DD` | **>12 月标"维护风险"** | GitHub repo |
| 最新发布版本 | `latest_release: vX.Y.Z @ YYYY-MM-DD` | >6 月留意 | GitHub releases / npm |
| 开放/总 issue 数比 | `issues: open/total` | — | GitHub repo |
| 许可证 | `license: MIT/AGPL/...` | 标注病毒式许可风险 | GitHub / 包元数据 |
| 前端 bundle 体积 | `bundle_size_gzip: KB` | — | Bundlephobia |
| 安装磁盘体积 | `install_size: MB` | — | Packagephobia |
| 主要维护方 | `maintainer: 公司/个人/基金会` | 单点风险 | GitHub / 项目主页 |

### TR-3: 笔记加代码片段段

子代理在三段 (来源/发现/缺口) 基础上多加一段:

````
## 代码片段
### [Xn] {库名} 基础调用
```{lang}
{≤15 行核心调用示例, 从官方文档抠}
```

### [Xm] {库名} 进阶/配置
{另一关键示例}
````

每候选 1-3 段, 整任务 ≤5 段。**只抠官方/作者写的代码**, 不抓二手教程的。

### TR-4: 报告必须含决策矩阵

`report-template.md` 的核心章节加表:

```
## 决策矩阵

| 维度 | {候选 A} | {候选 B} | {候选 C} |
|---|---|---|---|
| 维护活跃度 (最后提交) | 2026-04 ✅ | 2024-11 ⚠️ | 2026-05 ✅ |
| 星标 / 生态 | 45k | 12k | 88k |
| 许可证 | MIT ✅ | AGPL-3.0 ⚠️ | Apache 2.0 ✅ |
| 包体积 | 12KB | 45KB | 8KB |
| 学习曲线 | 低 | 中 | 高 |
| 团队栈契合 | {依据本地代码库} | ... | ... |
| 性能 (基准测试) | {数字 + 源} | ... | ... |
| 主要风险 | 单点维护者 | 许可证病毒式 | 学习成本 |

**推荐**: {候选} — {一句话理由 + 引用}
```

### TR-5: 加"实测建议"段

报告末尾 (在引文注册表前) 加:

````
## 实测建议 (最小验证)

最小验证命令:
```bash
{npm/pip/cargo install ...}
{最小 hello-world 代码或 cli 调用}
```

观察点:
- {点 1, 如"启动时间"}
- {点 2, 如"内存占用"}
- {点 3, 如"错误信息可读性"}

预计耗时: {N 分钟}
````

调研只是文献综述, 真正决策前必须有最小验证 — 本段告诉用户怎么花 30 分钟验证。

### TR-6: 本地代码库约束

起草前快速扫:
- `package.json` / `pyproject.toml` / `Cargo.toml` / `go.mod` 看现有栈
- 主要框架版本 (React 17 vs 19 限制库选择)
- 现有 ORM / HTTP 客户端 / 测试框架

报告"决策矩阵"的"团队栈契合"行据此填。**不要推荐与现有栈冲突的方案**, 即使技术更优。如必须推荐, 在风险列写明迁移成本。

### TR-7: 反模式 (技术调研特有)

- ✗ 用 web_search 拿 Medium/dev.to 二手博客当主源 (应去 context7 + GitHub)
- ✗ 推荐库不报最后提交日期
- ✗ 只看星标不看 issue 健康度 (50k 星但 3k 开放 issue = 维护塌方)
- ✗ 忽视许可证 (AGPL 进商业代码会引爆)
- ✗ 不收代码片段, 只给文字描述 (开发者要看 API 长啥样)
- ✗ 不读本地代码库就推荐方案 (推 React 库给 Vue 项目)
- ✗ 不查趋势源就推荐 (推 Tech Radar 已"Hold"圈的技术 = 给用户埋坑)
- ✗ 子代理对同一 URL 死磕重试 (走"抓取规约"段代理级联, 别烧 token)

### TR-8: 趋势源必查

凡技术调研 (命中"评估库选型 / 看技术风向"意图), 起草 (P4) 前必须查至少 1 个趋势源:

- **Thoughtworks Tech Radar** 最新卷 (thoughtworks.com/radar) — 看候选技术在 Adopt / Trial / Assess / **Hold** 哪一圈
- **State of JS / CSS / HTML / Devs** (Devographics 系列, stateofjs.com 等) — 看使用率 / 满意度 / 留存率
- **JetBrains Developer Ecosystem** / **GitHub Octoverse** — 年度宏观趋势

报告"决策矩阵"加 1 行: **趋势位置** — 标 `Adopt ✅` / `Trial 🟡` / `Assess 🟢` / `Hold ⚠️` (来自 Tech Radar) 或满意度百分比 + 调查源 (来自 State of XX)。

未查趋势源就推荐 = 反模式 (见 TR-7)。

## 状态回执约定

每阶段结束输出一行 `[Pn] ...` 形式回执给用户。用户随时可打断、改方向、跳过某阶段。失败 (web_search 不可用、子代理无回应、质量阀不达标) — 停下报告并等用户决定, 不要静默重试。

## P0: 范围 + 模式 + 意图识别

回问用户三件事 (一次提问) —

1. **深度** — 常规模式 (3 任务, 2-4K 字) / 深度模式 (4-5 任务, 4-7K 字)
2. **视角** — 调研立场 (如: 用户/开发者/采购方/学术综述)
3. **输出语言** — 中/英/双语对照

### 意图识别 (主代理自动判)

读用户原始问题, 命中下表关键词即贴标签。允许多标签 (主意图 + 副意图)。

| 意图标签 | 触发关键词 | 首选源 (TR-1 表对应) |
|---|---|---|
| 查 API/规范 | API / 语法 / spec / 规范 / 标准 / 怎么用 | MDN / W3C-TC39 / framework 自家文档 |
| 评估库选型 | 选哪个 / 对比 / A vs B / 迁移 / 选型 / 替代 | GitHub + context7 + Bundlephobia |
| 看技术风向 | 趋势 / 还能用 / 会不会过时 / 2025 / 2026 / 该不该上 | Tech Radar + State of XX |
| 实时趋势 | 最近 / 新出 / 发布 / 刚刚 / 这周 | HN + Lobsters + InfoQ |
| 问新问题 | 怎么实现 / 为啥不工作 / 求助 / 实战 | GitHub Discussions + Discord |
| 搜旧答案 | 报错 / error / 已知问题 / 历史 | Stack Overflow |
| 中文圈 | 中文教程 / 国内 / 汉化 / 中文社区 | 掘金精选 + InfoQ 中文 |

报告: `[P0] 主意图={X}, 副意图={Y/无}` — 让用户随时纠正。

### 模式决策

**默认**: 常规模式。

**升级到深度模式的触发** (任一命中即升):
- 用户明说"深度调研 / 综述 / 多源对比 / 技术选型 / 行业分析"
- 主题跨 3+ 子领域
- **意图反推**: 命中 2+ 意图且任一是"评估库选型"或"看技术风向" (这两个意图源多, 单一任务覆盖不全)

### 设定

- `时效基准` = 今天日期 (YYYY-MM-DD), 用于时效性核查
- `模式` = 常规 / 深度
- `主意图` / `副意图` = 上表标签
- 检查能力: web_search 必须可用 (不可用→停下问用户); 无子代理派遣能力则进入**降级顺序模式** (见 P2)

报告: `[P0] 模式={常规/深度}, 主意图={X}, 副意图={Y/无}, 时效基准={日期}, 子代理={是/否}`

## P1: 任务板

把研究问题拆 3-5 个任务, 每个由专家角色独立调查 —

```
任务 A — 角色: {如 "TLS 指纹研究员"}
         意图: {P0 识别的意图标签 — 如 "评估库选型"}
         首选源: {按意图从 TR-1 表自动填}
         目标: 一句话调查目标
         查询: 2-3 个预设搜索关键词
         深度: 深读 (全文 2-3 篇) / 速读 (摘要够用)
         输出: 任务 A 笔记

任务 B — ...
```

规则 —
- 任务彼此独立 + 来源多样
- **意图分流**: 每任务必有 1 个意图标签 (P0 表), 子代理按"首选源"先打头, 不够再扩 web_search
- **派遣分批**: 同时并发 ≤3, 超过 3 任务时分 2 批 (先 3, 等回完再剩余)
- 标记时效敏感声明
- 每任务必有"专家角色", 不写"通用搜索"
- 单任务可标"主意图 + 副意图"双标签, 两套源都覆盖 (源数会自动升到 5-8 条)

报告: `[P1] N 任务, 第 1 批 K 个派遣中`

## P2: 派遣 + 收笔记

并行派子代理, 加载 [references/subagent-prompt.md](references/subagent-prompt.md) 作为派遣模板。

子代理交付物按 [references/notes-format.md](references/notes-format.md) 格式 — 三段:
- **来源** — URL + 类型 + 日期 + 权威度 1-10 + **状态** (✓ 畅通 / ⚠ 代理 / ⚠ 快照 / ✗ 受限, 见"抓取规约"段)
  - **每任务源数**: 深度模式 4-8 条 / 常规模式 3-5 条 (聚合后达到 P3 全局阈值即可)
- **发现** — ≤15 条单句事实, 每条带源号 [An]
- **缺口** — 搜了但没找到 / 反向解释候选 / 时效风险 / **访问失败** (✗ 受限源)

**铁律** —
- 主代理在起草阶段**只读笔记**, 不把子代理或自己的原始搜索结果作为起草直接素材
- 子代理不许编 URL, 只用实际访问过的链接
- 不确定数字标 `[未核实]`
- 抓取失败按"抓取规约"段走代理级联, 不许死磕重试 WebFetch
- 每条来源必须带状态字段

### 降级顺序模式 (无子代理派遣能力时)

主代理串行扮演每个角色, 流程严格分段:

1. **调研窗口**: 当前任务的 web_search + WebFetch 全部执行
2. **蒸馏窗口**: 立刻写出 `任务 {id}` 笔记 (格式同上)
3. **关闭原始素材**: 笔记写完后**不回头查搜索结果**, 进入下一任务

铁律仍成立: 起草 (P4) 时只引用笔记, 不引用任何原始搜索。这是工作流分段而非物理隔离 — 关键是起草那一步只看蒸馏后的笔记, 防止主代理凭印象编内容。

报告每个任务: `[P2 任务 {id}] {N} 源, {M} 发现`

## 抓取规约 (通用, 不限技术调研)

子代理在 P2 抓取任何 URL 时遵守。WebFetch 命中 403 / 429 / Cloudflare / 登录墙 / 空响应时, **严禁死磕重试**, 按 [references/fetch-methods.md](references/fetch-methods.md) 走代理级联:

```
WebFetch → r.jina.ai → defuddle.md → archive.org → 标 ✗ 受限
```

GitHub 内容走专用通道 (raw + gh CLI), 不走代理级联。PDF 走 Jina 或本地 `pdftotext` / `marker_single`。

**源状态标记** (写入笔记来源段):

| 标签 | 含义 | 入注册表? | 权威度影响 |
|---|---|---|---|
| `✓ 畅通` | WebFetch 直拉成功 | ✓ | 无 |
| `⚠ 代理` | Jina / Defuddle 拿到 | ✓ | 自动 -1 |
| `⚠ 快照` | archive.org 拿到 | ✓ | 自动 -1, 标快照日期 |
| `✗ 受限` | 全链失败 | ✗ | 不入注册表, 进缺口段"访问失败" |

详细命令清单 + 服务可用性预警 + 反模式见 [references/fetch-methods.md](references/fetch-methods.md)。

## P3: 引文注册表 (锁源)

读完所有任务笔记 → 合并去重 → 统一编号 —

```
注册表

通过:
[1] 作者/机构 — 标题 | URL | 类型: 官方 | 日期: YYYY-MM-DD | 权威: 8 | 状态: ✓ 畅通 | 任务 A
[2] 作者/机构 — 标题 | URL | 类型: 社区 | 日期: YYYY-MM-DD | 权威: 5 | 状态: ⚠ 代理 (r.jina.ai) | 任务 B
[3] 作者/机构 — 标题 | URL | 类型: 行业分析 | 日期: YYYY-MM-DD | 权威: 6 | 状态: ⚠ 快照 (2025-03-08) | 任务 B
...

淘汰:
× 来源 | URL | 原因: 单源 + 权威<5 / 时效过期 / 重复 / ✗ 访问受限

统计: 通过={n}/{total}, 域名数={k}, 官方占比={pct}%, ⚠ 代理或快照={p}%
```

**质量阀** — 全部针对**聚合后的"通过"注册表**, 非单任务:

| 指标 | 深度模式 | 常规模式 |
|---|---|---|
| 通过源数 (去重后) | ≥10 | ≥6 |
| 唯一域名数 | ≥5 | ≥3 |
| 官方+学术 占比 | ≥30% | ≥20% |
| 单源最大占比 | ≤25% | ≤30% |
| 社区占比上限 | ≤30% | ≤40% |

不达标 — 停下报告给用户, 提议: ① 补 1-2 个针对性任务再跑 P2 / ② 降级到常规模式 / ③ 接受当前数据起草并标注信心度低。

**来源类型标签** — 官方 / 学术 / 行业分析 / 新闻 / 社区 / 其他

**P3 之后注册表锁死** — P4 起草不可加新源, 想到新事实没源就标 `[未核实]` 或删。

报告: `[P3] 通过 {n}/{total} 源, {k} 域名, 官方占比 {pct}%`

## P4: 大纲 + 起草

先建大纲 (主题先于任务序) —

```
## 章节 N
来源: [1][3][7] (跨 任务 A, 任务 B)
核心论断: ...
反向声明候选: ...
时效核查: 源日期 vs 时效基准
缺口: 哪些数据缺官方源
```

再按 [references/report-template.md](references/report-template.md) 起草。

**起草规则** —
- 每条事实带 `[n]`
- 数字/百分比必须有源, 否则 `[未核实]` 或删
- 每节末标**信心度**: 高/中/低 + 一句理由
- 冲突证据处插**反向声明** ("亦有 X 认为...")
- 主代理永不编 URL — 全从注册表取

报告: `[P4] {n}/{m} 章节, ~{words} 字`

## P5: 反向自审 + 核验 + 交付

自审 6 题 —
1. 结论可能错吗? 错在哪?
2. 哪些高影响声明只有 1 个源?
3. 哪些声明缺官方/学术背书?
4. 时效敏感声明的源日期 vs 时效基准?
5. 哪些主张对立解释成立?
6. ✗ 受限源是否影响关键声明? 报告里说了吗?

**强制找 ≥3 个问题** — 找不到就重审 (大概率自欺)。

**问题处置分流** (问题 ≠ 报告中的"核心争议") —
- 类型 A: 缺源/单源/时效过期 → 回头补源、降信心度、或删该声明
- 类型 B: 存在实质对立证据/解释 → 写入报告"核心争议"段保留
- 类型 C: 内部不一致/数字矛盾 → 修正文

只有类型 B 进入报告"核心争议"。A/C 是悄悄修。**不要为凑"核心争议 ≥3 条"硬塞类型 A 的问题**。

核验 —
- [ ] 每个 `[n]` 都在"通过"注册表里
- [ ] 抽 5+ 关键声明溯源到任务笔记
- [ ] 删/标无源声明
- [ ] "淘汰"源没复活
- [ ] 关键论点不靠单源
- [ ] ⚠ 转手源 (代理/快照) 占比 > 40% 时整体信心度下调
- [ ] ✗ 受限源进缺口段, 不在注册表; 影响章节在局限段明说

报告: `[P5] 发现 {n} 问题, 修正 {m} 处, 报告就绪`

交付时附 —
- 引文注册表 (放报告末尾或单独文件)
- 信心度概览
- 反向争议小节
- 时效基准日期

## 反模式

- ✗ 主代理直接读原始搜索结果 (应只读子代理笔记)
- ✗ 编造 URL / 凭印象写数字
- ✗ "淘汰"源在起草时偷偷复活
- ✗ 时效敏感声明缺日期
- ✗ 跳过 P5 自审或自审找 0 问题
- ✗ 用单源支撑高影响声明
- ✗ 子代理把任务搜索结果原样塞回主代理
- ✗ 模式选错 — 简单核对走常规模式即可, 不要事事深度模式
