# Search

派子代理搜资料 → 主代理整合成给同事看的调研笔记。**写工程笔记, 不写学术论文**。

## 流程 (2 步)

### 1. 派遣

主代理用 Agent tool 并发派子代理 (subagent_type=general-purpose), 模板见 [references/subagent.md](references/subagent.md)。

**派遣上限**:
- **默认 2 个子代理**
- **上限 3 个**, 仅当用户原题点名 3+ 候选, 或用户要求"全面对比 ≥ 5 个 X" 时允许
- 子任务按子代理 5 分钟内可完成的工作量拆分, 估计超时则任务过粗, 拆细后再派

**拆分规则**:
- 2 个子代理工作集互不重叠
- 合并完整覆盖用户原题

**反模式**:
- 同一主题派 4+ 子代理 (含同候选维度重复验证)
- 用户问 A, 顺带派子代理调研 B/C/D (周边发散)
- 同一候选派多个子代理深挖 (除非用户原题里出现"深挖 X")

### 2. 起草 (整合前先列 3 个清单)

收到子代理笔记后, 写报告前先列:

```
1. 重叠点清单 — 多份笔记讲了同一件事的, 每点只保留 1 处
2. 边缘点清单 — 与用户原题相关度低的内容, 整段删除
3. 优先级清单 — 直接影响选型决策的字段 (维护活跃 / 生态规模 / 学习成本) 排前, 次要细节后置
```

写报告时:
- 第一段直接答用户原题, 禁用 "本调研发现" / "经过研究" 开头
- 段末统一 `(参考 N)`, 禁止每句末挂 `[n]`
- 数据来源放报告末尾, **必须 `1. 2. 3.` 数字编号**, 1 行 1 源, **只列正文实际引用过的源, 上限 8 条**
- 写完后扫报告, 删除重复表达和冗余措辞

## 头部 3 行 (缺任一项则重写头部)

调研报告**前 3 行**必须按这个顺序:

```
1. 推荐: {首选 X, 一句理由}
2. 适用: {什么场景下首选 X 最稳}
3. 反推荐: {哪种场景下 X 反而是错的, 该选 Y}
```

**实例**:
```
推荐: Vue3 后台首选 vue-vben-admin v5 (32k 星 + 多 UI 内核可换)
适用: 想要开箱即用 + 国产生态 + 多 UI 适配 (EP / antd / naive 切换无痛)
反推荐: 不熟 Vue3 / 想脱国产生态锁, 选 shadcn-vue
```

**反模式**:
- "国内中后台默认 EP + pure-admin 或 vben v5; 看脸选 soybean; 想脱锁选 shadcn-vue" — 四档摆烂
- "Vue 是渐进式框架, 后台模板生态丰富..." — 不答原题, 先讲背景

候选间没有明显优势差距时, 用 "X / Y 二选一, X 适合 A, Y 适合 B" 句式, 仍给出明确推荐, 不并列 4+ 档。

## 输出风格

| 要 | 不要 |
|---|---|
| 头部 3 行: 推荐 / 适用 / 反推荐 | TL;DR / 摘要 / 本调研发现 / 四档摆烂 |
| 能成表的成表 (对比 ≥ 3 候选) | 三段并列描述同样字段 (a 怎样, b 怎样, c 怎样) |
| 人话段名: 推荐 / 对比 / 怎么试 / 数据来源 | 学术词: 决策矩阵 / 关键发现 / 引文注册表 / 附录 / 核心争议 / 局限 |
| 对比表 ≤ 5 列, 单元格 ≤ 20 字 | 6+ 列 / 单元格塞长描述 (如 "headless (Antd/MUI/Mantine/Chakra/shadcn 切换)") |
| 自然语言信心度 ("3 个官方源, 数据可靠") | "整体信心度: 中-高" 评级 |
| "数据截至 2026 年 5 月" 一句 | 列"模式 / 视角 / 主意图" 元数据 |
| 段落自然引文 (参考 1, 3) | `→ [N]` 箭头 |

## 结构

**主报告 ≤ 200 行**。超出按以下顺序处理:
1. 删除重叠点 / 边缘点 (起草前的 3 个清单已识别)
2. 仍超: 候选段落转对比表
3. 仍超: 派 1 个新子代理重新蒸馏现有笔记到 ≤ 200 行

**3 段必备**:
1. **头部 3 行** (推荐 / 适用 / 反推荐)
2. **对比表 或 正文** — 候选 ≥ 3 优先成表, < 3 用段落
3. **数据来源** (`1. 2. 3.` 编号, 1 行 1 源, 末尾一句可信度)

**可选段** (有实质内容才出, 否则整段省略):
- **怎么试** — ≤ 3 条, 每条**先场景再命令**, 不写纯命令清单。格式: `场景 / 适配类型 → 命令`。例: `用 antd 团队 → pnpm create umi@latest` 而非 `pnpm create umi@latest 选 ant-design-pro`
- **盲点** — ≤ 5 条单句, 仅记实际访问失败的源或确实没拿到的数据。**无内容不写, 禁止编造盲点凑形式**

### 数据来源段 (必备)

```
1. {作者} - {标题} - URL
2. {作者} - {标题} - URL
3. ...
```

**筛选规则**:
- **只列正文实际引用过的源** — 报告正文出现 `(参考 N)` 才进末尾段, 子代理拿了但主代理整合时没用上的源不列
- **上限 8 条** — 引用源超过 8 个时, 按 (权威度 × 时效性) 排序保留前 8

**必须数字编号**。报告正文的 `(参考 1, 3)` 必须严格对应到 `1.` `3.` 两个源。编号缺失或不对应则重写。

末尾用一句话描述信心度, 实例: "数据可信度: 大半官方源, 选型够用; 商业模板价格需自行复核当时定价页"。

## 抓取失败

WebFetch 失败按 [references/fetch.md](references/fetch.md) 执行代理级联。

## 不打断用户

派遣开始到生成报告之间, 主代理自动决策执行, 不向用户提问。抓取失败按级联处理, 全级联失败的源进报告末尾盲点段。

唯一例外: **主题歧义** (用户提到的名词指向多个独立对象, 如 "Cursor" 可指 IDE 或数据库游标) → 派遣前一次性问完所有歧义。

## 适合 / 不适合

| 适合 | 不适合 |
|---|---|
| 多源对比 (库对比 / 选型) | 单页问答 (用 WebFetch) |
| 综述 / 行业分析 / 政策研究 | 写文章 / 润色 |
| 调研报告需可追溯引用 | 单文件理解 (用 Read 工具) |

## 内部状态符号 (子代理笔记内部使用, 报告里禁用)

- 子代理笔记中标记抓取状态: `✓ 畅通` / `⚠ 代理` / `⚠ 快照` / `✗ 受限`
- 主代理起草时翻译成自然语言: ⚠ 代理 → "二手摘录", ✗ 受限 → 进盲点段
- 报告输出禁止出现 `✓` `⚠` `✗` 符号

## 自查 (输出报告前执行)

报告写完后, 扫描全文, 命中以下字面立即修改:

**字面黑名单**:
- 学术词: `TL;DR` / `决策矩阵` / `关键发现` / `引文注册表` / `核心争议` / `附录` / `本调研发现`
- 旧式引文: `→ [`
- 内部状态符号 (报告里禁用, 改自然语言): `✓ 畅通` / `⚠ 代理` / `✗ 受限`

**结构检查项** (任一项失败则修):
- 头部 3 行 (推荐 / 适用 / 反推荐) 是否完整? 缺任一项重写头部
- 候选 ≥ 3 时是否成对比表? 否则将段落转表
- 主报告行数 ≤ 200? 超出按"结构"段处理顺序压缩
- 数据来源段是否 `^1\.` `^2\.` 数字编号开头? 缺编号则重写
- 正文 `(参考 N)` 是否严格对应数据来源段编号? 不对应则修
- 数据来源段是否 ≤ 8 条且全部在正文有 `(参考 N)` 引用? 超量或有未引用源则按筛选规则砍
