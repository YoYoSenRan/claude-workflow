# Rules Generation

本参考用于生成 `.claude/rules/*.md`。rules 是持续生效的短规则，不是示例库，也不是长流程文档。

## 适合放入 rules 的内容

- 项目级硬约束；
- 高频、稳定、短小的开发规则；
- 路径匹配的目录规范；
- 命令选择的简短规则；
- 不要做的事项。
- 能防止模型高频改错的边界。

## 不适合放入 rules 的内容

- 长代码示例；
- 多个文件的证据清单；
- 复杂开发流程；
- 大段架构说明；
- 只有一个样例支撑的观察模式。
- 需要按步骤执行的任务流程；
- 只是描述项目里有什么的百科内容。

这些内容应该放到 skill 或 references。

## Rule 判定

候选进入 rule 前必须同时满足：

1. 持续生效，不依赖某一次任务；
2. 短，可以用几条约束表达；
3. 稳定，有文档或多个一致实现支撑；
4. 能约束模型行为，尤其是防止改错、跨层调用、错误命名、构建副作用、误改生成文件等。

如果需要长清单、示例、API 表、流程步骤，放 reference 或 skill，不放 rule。

## 通用规则模板

```markdown
# Code Style Rules

- 修改代码前先找同目录或同类型的相似实现。
- 默认复用现有导入顺序、命名方式、错误处理和类型组织。
- 不为单次使用新增抽象。
- 不引入新依赖，除非用户明确同意。
```

## 路径规则模板

```markdown
---
paths:
  - "src/modules/**/*.{ts,py,go,rs}"
---

# Module Rules

- 新模块入口必须复用当前项目已有组织方式。
- 写法以相邻同类实现为准。
- 行为变更需要补充适合当前项目的自动验证或人工核对步骤。
```

## 文件建议

| 文件 | 内容 |
|---|---|
| `00-project.md` | 项目级最高频规则 |
| `commands.md` | 常用命令选择和验证规则 |
| `code-style.md` | 通用代码风格和抽象边界 |
| `testing.md` | 测试和替代验证规则 |
| `<domain>.md` | 按项目实际领域命名，例如界面层、服务层、数据层等路径规则 |

只生成项目实际需要的 rules。
