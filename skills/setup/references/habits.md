# Coding Habits Profile

用这个文件执行编码习惯扫描。不要把它当说明文读；按步骤做，产出习惯矩阵，再决定生成哪些 rules、skills 和 references。

## 执行步骤

### 1. 选样例

先从文件索引和 rules 中找高频代码类型。常见类型包括组件、页面、API、store、hook、service、command、schema、config、test、script。

对每个高频类型执行：

1. 找 2-5 个同类文件。
2. 同时选择简单样例和复杂样例。
3. 优先选择当前仍在使用的文件。
4. 跳过生成文件、废弃目录、测试夹具和临时代码。
5. 如果只有 1 个样例，只记录为内部观察，不生成规则。

### 2. 提取字段

读取样例后，对每个维度填下面 5 个字段：

```markdown
| 维度 | 观察结果 | 证据文件 | 强度 | 建议落点 |
|---|---|---|---|---|
```

必须检查这些维度：

| 维度 | 提取动作 |
|---|---|
| 文件组织 | 写出新文件通常放哪里、目录如何分层、是否有聚合导出、文件名怎么命名 |
| 代码顺序 | 写出 import、常量、类型、状态、函数、生命周期、导出的大致排列 |
| 命名习惯 | 写出变量、函数、类型、组件、hook、store、API、文件名的命名模式 |
| 类型习惯 | 判断 interface / type 偏好、泛型写法、DTO / VO / Model 命名、内联类型边界 |
| 核心写法 | 写出 API 调用、状态管理、错误处理、异步流程、缓存、权限、路由、表单的项目写法 |
| 抽象边界 | 判断什么时候封装 hook / helper / service，什么时候直接写，哪些抽象应避免 |
| 注释习惯 | 判断注释语言、出现位置、是否解释业务原因、是否避免无意义注释 |
| UI / 组件 | 写出 props / emits / slots 顺序、样式组织、组件拆分、基础组件复用方式 |
| 数据流 | 写出页面取数、store 更新、接口数据转换、loading / error 放置方式 |
| 平台环境 | 写出多端、品牌、env、构建脚本、运行时差异的处理方式 |
| 验证习惯 | 写出 type-check、lint、test、build、手动验证如何选择 |
| 禁忌项 | 写出项目明显避免的依赖、目录、抽象、生成文件或写法 |

### 3. 判断强度

对每条观察只选一个强度：

| 强度 | 判断条件 | 允许动作 |
|---|---|---|
| 强规则 | 明确文档要求，或 3 个以上一致实现 | 可写入 `.claude/rules/*.md` 或领域 skill 执行规则 |
| 稳定习惯 | 2-5 个样例基本一致，但没有文档明说 | 写入 `context/references/habits.md` 或领域 `references/examples.md` |
| 内部观察 | 只有 1 个样例，或样例互相冲突 | 只放内部扫描账本，不默认展示给用户 |
| 不采用 | 来自生成文件、废弃目录、测试夹具、临时代码 | 不写入任何产物 |

### 4. 判断落点

按下面规则决定写到哪里：

| 情况 | 写入位置 |
|---|---|
| 跨领域、短、高频、稳定 | `.claude/rules/*.md` |
| 只影响某个领域的写法 | `.claude/skills/<domain>/references/examples.md` |
| 项目整体画像、习惯矩阵 | `.claude/skills/context/references/habits.md` |
| 命令和验证选择 | `.claude/skills/commands/SKILL.md` 或对应 references |
| 不稳定或证据少 | 只留内部扫描账本 |

不要把同一条内容同时大段写入 rule 和 skill。rule 写短规则；skill reference 写证据和例子。

### 5. 生成用户可见建议

用户默认只需要知道应该添加什么。输出时只列适合落地的内容：

```text
已分析内容：
- 编码习惯：<概括已形成的关键习惯，不列无效维度>

建议生成：
- <文件路径>：<要解决什么写法/流程问题>；<包含什么>
```

不要默认输出：

- 不适用的维度；
- 证据不足的维度；
- 弱观察；
- 完整扫描账本。

用户追问“怎么扫描的”或指出漏项时，再按内部扫描账本回答。
