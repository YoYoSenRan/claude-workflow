# Coding Habits Profile

读取时机：`setup` 需要从代码样例提取编码习惯、质量门或候选 rules / skills 时读取；只做路径索引时不要读取。

产出：习惯矩阵、强度判断和 rules / skills / references 落点建议。

用这个文件执行编码习惯扫描。不要把它当说明文读；按步骤做，产出习惯矩阵，再决定生成哪些 rules、skills 和 references。

## 执行步骤

### 1. 选样例

先从文件索引和 rules 中找高频代码类型。常见类型包括组件、页面、API、store、hook、service、command、schema、config、test、script。

对每个高频类型执行：

1. 找 2-5 个同类文件。
2. 同时选择简单样例和复杂样例。
3. 优先选择当前仍在使用的文件。
4. 跳过生成文件、废弃目录、测试夹具和临时代码。
5. 如果只有 1 个样例，只记录为内部观察，不生成规则。

### 2. 提取字段

读取样例后，对每个维度填下面 5 个字段：

```markdown
| 维度 | 观察结果 | 证据文件 | 强度 | 建议落点 |
|---|---|---|---|---|
```

必须检查这些维度：

| 维度 | 提取动作 |
|---|---|
| 文件组织 | 写出新文件通常放哪里、目录如何分层、是否有聚合导出、文件名怎么命名 |
| 代码顺序 | 写出 import、常量、类型、状态、函数、生命周期、导出的大致排列；如果项目顺序松散，建议按依赖关系优先、同层名称从短到长 |
| 命名习惯 | 写出变量、函数、类型、组件、hook、store、API、文件名的命名模式；如果项目习惯松散，建议采用简短语义化名称，函数用动词短语，布尔值用 is / has / can / should |
| 类型习惯 | 判断 interface / type 偏好、泛型写法、DTO / VO / Model 命名、内联类型边界 |
| 核心写法 | 写出 API 调用、状态管理、错误处理、异步流程、缓存、权限、路由、表单的项目写法 |
| 抽象边界 | 判断什么时候封装 hook / helper / service，什么时候直接写，哪些抽象应避免；同时检查函数复杂度、单一职责、高内聚低耦合 |
| 注释习惯 | 判断注释语言、出现位置、是否解释业务原因、是否避免无意义注释；TS 优先 TsDoc，JS 优先 JsDoc |
| UI / 组件 | 写出 props / emits / slots 顺序、样式组织、组件拆分、基础组件复用方式 |
| 样式规范 | 写出 CSS / SCSS / LESS / 组件 style 的命名、嵌套、属性顺序、单位、变量、响应式和状态类习惯；样式项目必须结合 `setup-styling` 结果 |
| 核心框架 | 识别项目是否有 CRUD、表格表单、装修引擎、配置 DSL、API factory、自研组件协议等核心框架；必须结合 `setup-framework` 结果写出定义入口、使用方式、配置协议、方法能力和新增步骤 |
| 数据流 | 写出页面取数、store 更新、接口数据转换、loading / error 放置方式 |
| 平台环境 | 写出多端、品牌、env、构建脚本、运行时差异的处理方式 |
| 验证习惯 | 写出 type-check、lint、test、build、手动验证如何选择 |
| 禁忌项 | 写出项目明显避免的依赖、目录、抽象、生成文件或写法 |

### 代码规范兜底基线

当项目样例互相冲突、缺少明确习惯，或者现状明显偏松散时，不要把坏习惯沉淀成规则。应使用下面的严格基线作为候选建议：

- 变量名称简短、语义化，禁止脱离上下文的 `data`、`temp`、`item1`、`obj`。
- 同一作用域内声明顺序先按依赖关系，再按名称长度从短到长，长度相同按字母序。
- import 分组：框架 -> 第三方 -> 项目别名 -> 相对路径 -> 类型导入 -> 样式 / 副作用导入；组内从短到长。
- export 先公开主 API，再辅助项；同层从短到长。
- 函数使用动词短语命名；布尔判断使用 `is` / `has` / `can` / `should`。
- 函数顺序是主流程在前、辅助函数靠近首次调用位置；同层从短到长。
- 函数只做一件事；超过一个职责、嵌套超过 3 层、条件分支难以命名时应拆分。
- 优先 early return，减少 `else` 包裹主流程。
- TS 使用 TsDoc，JS 使用 JsDoc；只解释业务原因、边界和非显然约束。
- 高复用逻辑才抽 helper；单次使用不提前抽象。

### 样式规范兜底基线

当项目存在 CSS / SCSS / LESS / 组件 style，但样例互相冲突或缺少明确习惯时，不要沉淀松散写法。应使用下面的严格基线作为候选建议：

- 新增复杂样式优先使用 SCSS 或 LESS 嵌套；简单局部样式写在组件作用域内。
- 类名采用 BEM：`block`、`block__element`、`block--modifier`；状态类使用 `is-*` 或 `has-*`。
- 嵌套不超过 3 层；优先使用 `&__element`、`&--modifier`，避免长后代链。
- 同一选择器内属性按属性名称长度从短到长排列，长度相同按字母序。
- 避免 `!important`、过深选择器、无语义类名和只描述颜色 / 方位的类名。
- 颜色、字号、间距、圆角、层级优先使用项目已有变量、mixin 或 token。
- 单位跟随项目和平台；uni-app 优先尊重 rpx，Web 项目优先尊重已有 px / rem 策略。
- 响应式、暗色模式、平台差异和第三方覆盖必须复用项目已有入口。

### 3. 判断强度

对每条观察只选一个强度：

| 强度 | 判断条件 | 允许动作 |
|---|---|---|
| 强规则 | 明确文档要求，或 3 个以上一致实现 | 可写入 `.claude/rules/*.md` 或领域 skill 执行规则 |
| 稳定习惯 | 2-5 个样例基本一致，但没有文档明说 | 写入 `.claude/skills/<skill>/references/<file>.md`（文件名按内容定，例如 `habits.md`、`examples.md`） |
| 内部观察 | 只有 1 个样例，或样例互相冲突 | 只放内部扫描账本，不默认展示给用户 |
| 不采用 | 来自生成文件、废弃目录、测试夹具、临时代码 | 不写入任何产物 |

### 4. 判断落点

按下面规则决定写到哪里：

| 情况 | 写入位置 |
|---|---|
| 跨领域、短、高频、稳定 | `.claude/rules/*.md` |
| 只影响某个领域的写法 | `.claude/skills/<domain>/references/examples.md` |
| 项目整体画像、习惯矩阵 | `.claude/skills/<skill>/references/habits.md`；无 skill 归属则降级 internal |
| 命令和验证选择 | 命令 reference、验证 rule；只有存在任务流程时才生成验证类 skill |
| 不稳定或证据少 | 只留内部扫描账本 |

不要把同一条内容同时大段写入 rule 和 skill。rule 写短规则；skill reference 写证据和例子。

### 5. 生成用户可见建议

用户默认只需要知道应该添加什么。输出时只列适合落地的内容：

```text
已分析内容：
- 编码习惯：<概括已形成的关键习惯，不列无效维度>

建议生成：
- rules：
  | 路径 | 使用场景 / 作用 | 内容概要 |
  |---|---|---|
- skills：
  | 路径 | 使用场景 / 作用 | 内容概要 |
  |---|---|---|
- references：
  | 路径 | 承载内容 | 用途 |
  |---|---|---|
```

不要默认输出：

- 不适用的维度；
- 证据不足的维度；
- 弱观察；
- 完整扫描账本。
- 未采纳项清单。

用户追问“怎么扫描的”或指出漏项时，再按内部扫描账本回答。
