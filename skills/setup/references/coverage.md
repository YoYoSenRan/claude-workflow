# Coverage Matrix

本参考用于防止 setup 扫描不全或过度自信。生成任何 rules 或 project skills 前，必须先形成覆盖矩阵；面向用户输出时，应优先表达“已分析内容”和“建议生成内容”，不要把大量“不生成”项作为主要结果。

## 扫描原则

- 先做全项目广度索引，再做领域深度取样。
- 不因为先读到 UI、API 或测试样例就提前生成结论。
- 每个候选 project skill 都必须有领域存在证据和样例证据。
- 证据不足时写入 setup report，不要强行生成。
- 覆盖矩阵是内部决策依据；用户可读输出应先给“已分析内容”和“建议生成内容”。
- 已有 rules 是覆盖证据，不是跳过 project skills 的理由。
- 高频开发领域存在且证据充分时，建议动作应是“生成/刷新对应 project skill”，不是“维持”或“作为不生成理由”。

## 覆盖等级

| 等级 | 含义 | 允许动作 |
|---|---|---|
| 充分 | 有明确文档规则，或至少 2-5 个一致样例 | 可生成 rule 或 project skill |
| 部分 | 领域存在，但样例少或模式不一致 | 可生成 project skill，但必须标风险；不要写强 rules |
| 不足 | 只有目录名、单个文件或无法判断模式 | 不生成；写入未覆盖 |

## 覆盖矩阵模板

```markdown
| 领域 | 是否存在 | 证据 | 样例数 | 当前承载 | 建议动作 | 覆盖等级 | 风险 |
|---|---|---|---:|---|---|---|---|
| <项目领域> | 是 | <目录/rule/脚本> | 3 | <当前承载> | 生成 project-<项目领域名> | 充分 | - |
| <命令> | 是 | package.json scripts | 1 | package.json | 生成 project-commands | 充分 | 命令入口分散 |
| <弱证据领域> | 是 | <单一目录> | 1 | - | 写入 project-context，不生成独立 skill | 不足 | 样例少 |
```

用户可读输出不要逐条列出所有“未生成 project-*”。只有两类情况需要说明：

- 该领域会影响用户决策，例如没有测试体系；
- 该领域证据不足但用户很可能期待，例如数据库、API、UI 等核心领域。

## 必须报告的风险

- 代码样例风格不一致；
- README、CLAUDE.md、AGENTS.md 与实际代码冲突；
- package scripts 和 CI 命令不一致；
- 测试框架存在但测试样例很少；
- 历史旧目录可能不代表当前规范；
- 发现领域入口但没有足够样例。
