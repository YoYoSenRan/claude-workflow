# Coverage Matrix

按本文件形成内部覆盖矩阵，防止 setup 扫描不全或过度自信。用户默认只看“已分析内容”和“建议生成内容”。

## 执行顺序

1. 先建立全项目文件索引。
2. 再读取 rules、docs、配置和命令入口。
3. 然后按 `habits.md` 建立编码习惯矩阵。
4. 再识别领域并读取代表实现。
5. 最后形成内部覆盖矩阵。

执行时遵守：

- 广度索引必须来自目标项目文件系统，不能用已注入上下文替代实际读取。
- 不因为先读到某类样例就提前生成结论。
- 每个候选 skill 都必须有证据、明确任务触发、执行顺序和生成价值。
- 证据不足时不要强行生成；默认从用户候选输出中省略。
- 已有 rules 是覆盖证据，不是自动生成或跳过 skill 的理由。
- 高频开发领域存在且证据充分时，仍需按产物决策协议判断：rule / skill / reference / internal。
- 用户可读输出默认不展示完整扫描账本；扫描账本用于内部判断和用户追问时核对。
- 如果只发现路径但没有读取内容，不能算作已检查。
- 如果只读了 rules，没有读对应代表源码，不能把该领域标为“充分”，最多标为“部分”。

## 覆盖等级

| 等级 | 含义 | 允许动作 |
|---|---|---|
| 充分 | 有明确文档规则，或至少 2-5 个一致样例 | 可生成 rule、skill 或 reference |
| 部分 | 领域存在，但样例少或模式不一致 | 优先写 reference 或 internal；补足任务触发和一致样例后再生成 skill |
| 不足 | 只有目录名、单个文件或无法判断模式 | 只进入内部账本；默认不展示 |

## 覆盖矩阵模板

```markdown
| 领域 | 是否存在 | 证据 | 样例数 | 当前承载 | 建议动作 | 覆盖等级 | 内部备注 |
|---|---|---|---:|---|---|---|---|
| <领域> | 是 | <目录/rule/脚本> | 3 | <当前承载> | 生成 <领域名> / 写 reference / 写 rule | 充分 | 按生成价值判断 |
| <命令> | 是 | package.json scripts | 1 | package.json | 写命令 reference；必要时生成验证类 skill | 充分 | 命令入口分散 |
| <弱证据领域> | 是 | <单一目录> | 1 | - | 写入 domains reference，不作为独立 skill 候选 | 不足 | 样例少 |
```

用户可读输出不要逐条列出未采纳领域或证据较弱的候选。只有两类情况需要说明：

- 该缺口会阻塞本次 setup 写入；
- 该缺口会导致用户按建议生成后明显误用。

## 内部阻塞检查

生成候选前检查这些情况。默认不展示；只有会阻塞写入或导致明显误用时，才在用户输出中说明：

- 代码样例风格不一致；
- README、CLAUDE.md、AGENTS.md 与实际代码冲突；
- package scripts 和 CI 命令不一致；
- 测试框架存在但测试样例很少；
- 历史旧目录可能不代表当前规范；
- 发现领域入口但没有足够样例。

## 内部扫描账本模板

```text
已检查：
- 文件索引：顶层目录 / .claude / docs / scripts / tests / CI / src / packages / apps
- 项目入口：CLAUDE.md / README.md / docs/...（已读取 / 不存在 / 仅发现路径）
- Claude 内容：.claude/rules/*.md / .claude/skills/...（已读取 / 不存在 / 仅发现路径）
- 包配置：package.json / tsconfig / lint / format / CI（已读取 / 不存在 / 仅发现路径）
- 代码领域：src/... / packages/... / apps/...（已读取样例 / 仅发现路径）
- 习惯维度：写法风格 / 代码顺序 / 命名 / 类型 / 核心写法 / 抽象 / 注释 / 数据流 / 验证（已读取样例 / 证据不足）
- 验证入口：scripts / test / lint / build
- 代表样例：<领域> -> <已读取文件1>, <已读取文件2>

未检查 / 证据不足：
- <路径或领域>：<原因>
```

不要写“完整扫描”或“全部覆盖”这类不可核对结论。默认用户输出不贴完整账本；用户追问扫描过程时再给。
