# 刷新已有配置

读取时机：`setup` 发现目标项目已有 `.claude/rules/`、`.claude/skills/` 或 `CLAUDE.md` 时读取；首次初始化的项目不要读取。

产出：刷新策略、冲突处理规则、保留/更新/删除判断。

## 何时进入刷新模式

扫描第 1 步发现以下任一情况时，进入刷新模式：

- `.claude/rules/` 下有文件
- `.claude/skills/` 下有文件
- 项目根或 `.claude/` 下有 `CLAUDE.md`

刷新模式不是"重新生成"，而是"对照当前代码检查已有配置，该保留的保留，该更新的更新，该清理的指出来"。

## 分类已有内容

先把已有的每个 rule / skill / reference 分成两类：

**手动创建的** — 用户自己写的，不是 setup 生成的。识别方法：
- 文件内容有明显的手写风格（个性化表述、具体业务术语）
- 没有 setup 生成的标记或模板痕迹
- 用户确认是手写的

**自动生成的** — 之前 setup 生成的。识别方法：
- 结构符合 setup 模板（habits 矩阵、examples 模板等）
- 有 setup report 引用

分不清的，当作手动创建的对待——宁可多保留也不要误删。

## 对照检查

对每个已有的 rule / skill：

### 1. 证据是否还在

rule 或 skill 引用的文件、目录、模式还存在吗？

- **都在** → 继续检查一致性
- **部分不在** → 标记为"需要更新"，向用户说明哪些引用过时了
- **全部不在** → 标记为"建议删除"，向用户确认

### 2. 内容是否还一致

rule 描述的规范，跟当前代码的实际写法一致吗？

- **一致** → 保留不动
- **代码偏离了 rule** → 两种可能：
  - rule 是对的，代码有问题 → 保留 rule，提醒用户代码偏离
  - 代码已经演化，rule 过时了 → 建议更新 rule
  - 分不清哪种 → 向用户展示差异，让用户决定
- **rule 描述太粗/太细** → 建议调整粒度

### 3. 有没有新发现

扫描发现了已有 rule 没覆盖的模式：

- 如果这个模式符合 rule 生成标准（短、稳定、3+ 样本）→ 建议新增 rule
- 如果已有 rule 可以扩展覆盖 → 建议更新已有 rule
- 不要把新发现跟已有 rule 合并成一个大杂烩 → 保持每个 rule 职责单一

## 处理规则

| 情况 | 操作 |
|---|---|
| 手动创建 + 内容仍准确 | 保留不动，不要碰 |
| 手动创建 + 部分过时 | 展示哪些过时了，建议用户自己更新 |
| 手动创建 + 完全过时 | 建议用户删除，但不要自动删 |
| 自动生成 + 内容仍准确 | 保留不动 |
| 自动生成 + 需要更新 | 展示差异，用户确认后更新 |
| 自动生成 + 完全过时 | 展示原因，用户确认后删除 |
| 新发现 + 符合生成标准 | 按正常 setup 流程建议新增 |

**核心原则：不自动覆盖、不自动删除。所有变更都要用户确认。**

## 向用户展示刷新结果

```text
已有配置检查结果：

保留不动（N 个）：
- rules/api.md — 内容与代码一致
- rules/code-style.md — 内容与代码一致

建议更新（M 个）：
- rules/routing.md — 路由注册方式已从 X 改为 Y
- skills/crud/references/components.md — 新增了 3 个组件未收录

建议删除（K 个）：
- rules/deprecated-pattern.md — 引用的目录已不存在

建议新增（L 个）：
- rules/error-handling.md — 发现统一的错误处理模式（5 个样本）

需要确认后执行。
```

## 不要做的事

- 不要把手动写的 rule 重新生成成 setup 风格 — 保持用户的原始表述
- 不要因为"风格不统一"就重写已有 rule — 内容准确比风格统一重要
- 不要默认全量重新生成 — 刷新是增量操作
- 不要把"代码偏离了 rule"自动当作"rule 过时了" — 可能是代码有问题
