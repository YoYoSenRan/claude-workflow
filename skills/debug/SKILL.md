## 子代理辅助模式

被派遣排查问题时，做只读根因调查。只返回复现情况、证据、根因假设和建议验证；不修改文件、应用修复或宣布问题已修好。

# 系统化排错

随机修补会掩盖根因。debug 先证明问题从哪里来，再做最小修复。

<HARD-GATE>
阶段 1 没完成前，禁止提出修复方案。

禁止跳阶段、同时改多处、用“应该是”替代证据。

连续 3 次修复失败后，必须停下质疑架构，不允许试第 4 次。
</HARD-GATE>

## 何时使用

测试 / 构建 / 运行时失败；行为和预期不一致；性能或集成异常；`execute` 执行计划时遇到无法解释的失败。

不使用：新功能开发（走 `think` / `plan`）；纯重构且无失败信号；预防性加固；明确单字符笔误且用户已给准确位置。

## 四阶段流程

### 阶段 1：根因调查

修复前必须完成：

1. **读错误** — 完整读错误信息、调用栈、文件路径、行号、错误码。
2. **复现** — 确认能否稳定触发；不能复现就收集更多证据，不猜。
3. **看最近变化** — `git diff`、最近提交、配置和依赖变化。
4. **追数据流** — 从报错点向上追，找坏值或坏状态从哪里进入。
5. **多组件诊断** — CI / 构建 / API / 数据库链路，在边界处加证据或检查输出。

输出（缺这两项不进 阶段 2）：

```text
根因假设：<具体原因>
证据：<错误 / 代码 / 复现步骤 / 最近变化>
```

### 阶段 2：模式分析

找参照，不凭感觉：找同仓库能工作的相似实现 → 完整读（不略读）→ 列工作版本和失败版本的差异 → 检查依赖、配置、输入数据和调用约定。

### 阶段 3：单一假设验证

一次只验证一个假设：写下 `我认为根因是 X，因为 Y` → 做最小验证（只改一个变量或加一个诊断点）→ 命中进 阶段 4；失败回 阶段 1 / 2 形成新假设。不要在失败假设上叠加更多修复。

### 阶段 4：实现修复

1. **写失败测试或最小复现** — 有测试框架就写自动测试，没有就写一次性复现命令或脚本；必须先看到它失败。
2. **单一修复** — 只改根因所需位置，不顺手重构或修旁边问题。
3. **新鲜验证** — 本轮重跑相关测试或复现命令，读完整输出，证明：原始症状不再出现、相关测试通过、没有明显新增失败。
4. **失败处理** —
   - 每次失败后，显式记录第几次尝试、当时假设和结果（写进回复或 TodoWrite），避免跨轮或 compact 后丢失计数。
   - 第 1-2 次失败：带新证据回到 阶段 1。
   - 第 3 次失败：进入 阶段 4.5。

### 阶段 4.5：质疑架构

连续 3 次修复失败，停止继续试。架构问题信号：每次修复都暴露新的共享状态或耦合点；修一处必须大规模改另一处；补丁越来越像掩盖症状；同类失败在多个位置重复出现。

此时输出：

```text
当前不像单点 bug，更像架构/边界问题。
已尝试：
1. ...
2. ...
3. ...
建议回到 think / plan 重新设计修复方案。
```

## 与其他 skill 的关系

- 修复完成后转 `verify` 做最终完成门：说明运行了什么命令、关键输出、是否还有失败或未验证项。
- 单点 bug 在本 skill 内完成修复和验证；需要跨多文件或改架构 → 停，转 `think` / `plan`。
- `execute` 某步骤失败且原因不明 → 转 `debug`；debug 不直接改执行计划，计划错应回 `plan`。

## 警示信号

| 念头 | 现实 |
|---|---|
| "先加个判断试试" | 没根因就是症状补丁。 |
| "这个错误很明显" | 明显也要有证据。 |
| "同时改几处省时间" | 会失去变量隔离。 |
| "测试太麻烦，手动看一下" | 没有可重复验证，修复不可信。 |
| "这次失败可能是环境问题" | 没证据不能归因环境。 |
| "已经试了三次，再试一次" | 第 4 次前必须质疑架构。 |

## 用户纠偏信号

用户出现这些反馈时，立刻回到 阶段 1："你怎么知道？" / "有证据吗？" / "是不是没看到输出？" / "别猜了。" / "深度想想。" / "这到底是不是根因？"

## 交付格式

```text
根因：...
修复：...
验证：运行 <命令>，结果 <关键输出/通过条件>。
未验证/风险：...
```

没有验证时，不允许说“已修复”。
