# 系统化调试

## 概述

随意修复浪费时间，并制造新的 bug。临时补丁会掩盖底层问题。

**核心原则：** 在尝试修复之前，永远先找到根本原因。仅修复表面症状即失败。

**违背该流程的字面，就是违背调试的精神。**

## 铁律

```
NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST
```

如果你没有完成阶段 1，就不能提出修复方案。

## 何时使用

适用于任何技术问题：
- 测试失败
- 生产环境中的 bug
- 意外行为
- 性能问题
- 构建失败
- 集成问题

**尤其在以下情况使用：**
- 时间压力大（紧急情况让人忍不住乱猜）
- 看起来"只需一个小修复"很显然
- 你已经尝试过多次修复
- 上一次修复没生效
- 你并未完全理解问题

**不要因为以下原因跳过：**
- 问题看起来很简单（简单 bug 也有根本原因）
- 你赶时间（赶时间必然返工）
- 经理要求立刻修好（系统化比胡乱折腾更快）

## 四个阶段

你必须完成每个阶段，才能进入下一个。

### 阶段 1：根本原因调查

**在尝试任何修复之前：**

1. **仔细阅读错误信息**
   - 不要跳过错误或警告
   - 它们往往包含确切的解决方案
   - 完整阅读堆栈跟踪
   - 记录行号、文件路径、错误码

2. **稳定复现**
   - 你能可靠地触发它吗？
   - 确切的步骤是什么？
   - 每次都会发生吗？
   - 如果无法复现 → 收集更多数据，不要乱猜

3. **检查近期改动**
   - 什么改动可能导致这个问题？
   - Git diff、近期提交
   - 新依赖、配置变更
   - 环境差异

4. **在多组件系统中收集证据**

   **当系统由多个组件构成时（CI → 构建 → 签名，API → 服务 → 数据库）：**

   **在提出修复之前，添加诊断埋点：**
   ```
   对每个组件边界：
     - 记录进入组件的数据
     - 记录退出组件的数据
     - 验证环境变量 / 配置的传递
     - 检查每一层的状态

   先运行一次收集证据，看清楚它在哪里断裂
   然后分析证据找出失败组件
   然后再去调查这个具体组件
   ```

   **示例（多层系统）：**
   ```bash
   # Layer 1: Workflow
   echo "=== Secrets available in workflow: ==="
   echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

   # Layer 2: Build script
   echo "=== Env vars in build script: ==="
   env | grep IDENTITY || echo "IDENTITY not in environment"

   # Layer 3: Signing script
   echo "=== Keychain state: ==="
   security list-keychains
   security find-identity -v

   # Layer 4: Actual signing
   codesign --sign "$IDENTITY" --verbose=4 "$APP"
   ```

   **这能揭示：** 哪一层失败（secrets → workflow ✓，workflow → build ✗）

5. **追踪数据流**

   **当错误深埋于调用栈中：**

   - 错误值从哪里产生？
   - 是什么用错误值调用了它？
   - 一直向上追，直到找到源头
   - 在源头修复，而不是在症状处修复

### 阶段 2：模式分析

**在修复之前找到模式：**

1. **找到可工作的样例**
   - 在同一代码库中找到类似的可工作代码
   - 与坏的代码相似且能工作的是什么？

2. **对照参考实现**
   - 如果你要实现某个模式，**完整**阅读参考实现
   - 不要略读 —— 每一行都要读
   - 在套用前完全理解该模式

3. **找出差异**
   - 可工作的代码与坏的代码有什么不同？
   - 列出每一处差异，再小也要列
   - 不要假设"这不可能有影响"

4. **理解依赖**
   - 这块代码还需要哪些其他组件？
   - 需要什么设置、配置、环境？
   - 它做了哪些假设？

### 阶段 3：假设与测试

**科学方法：**

1. **形成单一假设**
   - 清楚陈述："我认为 X 是根本原因，因为 Y"
   - 写下来
   - 要具体，不要含糊

2. **以最小变更测试**
   - 做**最小**的改动来验证假设
   - 一次只动一个变量
   - 不要同时修多处

3. **继续前先验证**
   - 生效了？是 → 进入阶段 4
   - 没生效？形成**新**假设
   - 不要在原有修复上叠加更多修复

4. **当你不知道时**
   - 说"我不理解 X"
   - 不要假装知道
   - 求助
   - 再研究

### 阶段 4：实施

**修复根本原因，而不是症状：**

1. **创建失败的测试用例**
   - 最简单的复现
   - 尽可能用自动化测试
   - 没有框架就写一次性测试脚本
   - 在修复**之前**必须有
   - 按测试先行流程写出合格的失败测试

2. **实施单一修复**
   - 针对已识别的根本原因
   - 一次只改一个地方
   - 不要"顺便"做改进
   - 不要捆绑重构

3. **验证修复**
   - 测试现在通过了吗？
   - 没有其他测试被弄坏吧？
   - 问题是否真的解决了？

4. **如果修复未生效**
   - 停下
   - 数一数：你尝试过多少次修复了？
   - 如果 < 3：返回阶段 1，结合新信息重新分析
   - **如果 ≥ 3：停下并质疑架构（下方第 5 点）**
   - 在没有架构层面讨论之前，不要尝试第 4 次修复

5. **如果 3 次以上修复失败：质疑架构**

   **预示架构问题的模式：**
   - 每次修复都暴露出在不同位置的新共享状态 / 耦合 / 问题
   - 修复需要"大规模重构"才能落地
   - 每次修复都在别处产生新症状

   **停下并质疑基本面：**
   - 这个模式从根本上合理吗？
   - 我们是不是"纯粹靠惯性硬撑"？
   - 是否应该重构架构，而非继续修症状？

   **在尝试更多修复之前，与你的用户讨论**

   这不是一个假设失败 —— 这是错误的架构。

## 警示信号 —— 停下并按流程来

如果你发现自己在想：
- "先快速修一下，回头再查"
- "试着改一下 X 看看行不行"
- "一次加多个改动，跑一下测试"
- "跳过测试，我手动验证一下"
- "可能是 X，让我先把它修了"
- "我不完全理解，但这可能行"
- "模式说是 X，但我会有所变通地套用"
- "主要问题如下：[未经调查就列出修复清单]"
- 在追踪数据流之前就给出方案
- **"再试一次修复"（已经尝试过 2 次以上时）**
- **每次修复都在不同位置暴露新问题**

**这些**全都**意味着：停下，回到阶段 1。**

**如果 3 次以上修复失败：** 质疑架构（见阶段 4.5）

## 来自用户的"做错了"信号

**留意以下转向信号：**
- "Is that not happening?"（那件事没发生吗？） —— 你没验证就做了假设
- "Will it show us...?"（这会让我们看到……吗？） —— 你本应加上证据收集
- "Stop guessing"（别瞎猜了） —— 你在没理解的情况下提方案
- "Ultrathink this"（深度思考一下） —— 质疑基本面，而不只是症状
- "We're stuck?"（我们卡住了？语气沮丧） —— 你的方法不奏效

**当你看到这些时：** 停下，回到阶段 1。

## 常见的自我合理化

| 借口 | 现实 |
|------|------|
| "问题简单，不需要流程" | 简单问题也有根本原因。流程对简单 bug 也很快。 |
| "紧急情况，没时间走流程" | 系统化调试比瞎猜试错**更快**。 |
| "先试试这个，回头再查" | 第一次修复定下了基调。一开始就做对。 |
| "我先确认修复有效，再写测试" | 没测试过的修复留不下来。测试先行才能证明。 |
| "一次性多修几处省时间" | 无法隔离起作用的是哪一处。会引入新 bug。 |
| "参考太长，我适配一下这个模式" | 部分理解必然带来 bug。完整读完它。 |
| "我看到问题了，让我把它修了" | 看到症状 ≠ 理解根本原因。 |
| "再试一次修复"（已经失败 2 次以上） | 3 次以上失败 = 架构问题。质疑模式，不要再修一次。 |

## 速查

| 阶段 | 关键活动 | 成功标准 |
|------|----------|----------|
| **1. 根本原因** | 阅读错误、复现、检查改动、收集证据 | 理解**什么**和**为什么** |
| **2. 模式** | 找到可工作的样例、对比 | 识别差异 |
| **3. 假设** | 形成理论、最小化测试 | 验证或形成新假设 |
| **4. 实施** | 创建测试、修复、验证 | bug 解决，测试通过 |

## 当流程显示"没有根本原因"

如果系统化调查显示问题确实只是环境性的、与时序相关的或外部的：

1. 你已经完成了流程
2. 记录你调查过的内容
3. 实施合适的处理（重试、超时、错误信息）
4. 添加监控 / 日志以备将来调查

**但是：** 95% 的"没有根本原因"的案例其实是调查不彻底。

## 相关技能

- **claude-workflow:test** —— 用于创建失败的测试用例（阶段 4，步骤 1）
- **claude-workflow:verify** —— 在宣称成功之前验证修复是否真的有效

## 真实影响

来自调试会话：
- 系统化方法：15-30 分钟修好
- 随意修复方法：2-3 小时折腾
- 一次成功率：95% vs 40%
- 引入的新 bug：接近零 vs 常见
