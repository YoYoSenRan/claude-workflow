# 系统化排错

随机改代码碰运气浪费时间且制造新 bug。先找根因, 再动手。**症状级修复 = 失败**。

<HARD-GATE>
没完成 Phase 1 (读错误 + 复现 + 看 recent 改动) 之前, **禁止**提出任何修复方案 (包括 "我试试改 X" 类口头试探)。

禁止跳阶段、禁止"先快速试一下"、禁止同时改多处。

3 次修复失败 → 停, **质疑架构**, 不是再试第 4 次。
</HARD-GATE>

## 反模式: "这 bug 简单, 不用走流程"

最常掉的坑。简单 bug 也有根因, 流程对简单 bug 很快, 不耽误事。**"快速试一下" = 50% 错方向, 1 小时 thrash**。

## 4 阶段流程

每阶段必须完成才进下一阶段, 不可跳。

### Phase 1: 根因调查

动手前必做:

1. **认真读错误** — stack trace 全读, 行号 / 文件 / 错误码 / 关键字别跳过
2. **复现** — 能稳定触发吗? 步骤是啥? 不能复现 → 收集更多数据, **不准猜**
3. **看 recent 改动** — `git log` / `git diff` 最近几次; 新依赖; 配置变化
4. **多组件系统加诊断点** — CI → build → sign 这类多层链路, 给每个边界加 log, 跑一次看哪层断
5. **倒推数据流** — 错误深在调用栈, 沿调用链回溯到坏值起点, 在源头修不在表面修

输出: 一句话写下根因 + 证据。

### Phase 2: 模式分析

找参照, 不凭感觉:

1. **找工作样本** — 同代码库里相似的、能跑通的代码, 拿来对比
2. **完整读参考实现** — 不准 skim, 一行行读
3. **列差异** — 工作 vs 坏掉的, 列**所有**差异 (哪怕"那不可能影响")
4. **看依赖** — 需要什么 config / env / 依赖? 假设了什么?

### Phase 3: 假设 + 最小验证

科学方法:

1. **写下单一假设** — "我认为根因是 X, 因为 Y" (具体, 非含糊)
2. **最小测试** — 改**一处**验证假设, 一次只动一个变量
3. **验证** — 中? → Phase 4。没中? → 形成**新**假设, 不要在错的修上叠加
4. **不懂就说不懂** — "我不理解 X"。别装懂, 别瞎试。问用户或停下查资料

### Phase 4: 实现修复

修根因, 不修症状:

1. **先写失败测试** — 最简复现, 自动化优先 (有测试框架就写, 没有写个一次性脚本)。测试**必须先红**, 没看到红 = 没验证过它真测这 bug
2. **单一修复** — 一次改一处, 不"顺手优化"
3. **跑验证** — 在本轮重跑测试命令, 读完整输出, 看 0 failed + exit 0 + bug 触发步骤已不复现。没新鲜证据不准声明修好
4. **修没成?** — 数失败次数:
   - <3 次 → 回 Phase 1 重新分析 (带新信息)
   - **≥3 次 → 停, 不试第 4 次。质疑架构** (下面 4.5)

### Phase 4.5: 3+ 失败 → 质疑架构

**架构问题的信号**:

- 每次修都暴露**新位置**的共享 state / 耦合
- 修一处要"大规模重构"才行
- 每次修创造新症状在别处

**停下, 不要试第 4 次修**, 问基础:

- 这模式根本对吗?
- 是不是只因"惯性"在硬撑?
- 该重构架构而非继续修症状?

跟用户讨论再动手。这**不是**假设错, 是架构错。

## 流程图

```
看到 bug / 报错
   ↓
Phase 1: 根因调查
   ├─ 读错误 (stack trace 全读)
   ├─ 复现 (稳定能触发)
   ├─ 看 recent 改动
   └─ 倒推数据流到源头
   ↓
有根因假设?
   ↓ 是                  ↓ 否
Phase 2: 找工作样本对比  收更多证据 (加 log)
   ↓                     ↓ 回 Phase 1
Phase 3: 形成假设 + 最小验证
   ↓
中?
   ↓ 是                  ↓ 否
Phase 4: 写失败测试       回 Phase 3 新假设
+ 单一修复 + 验证          (不准在错修上叠加)
   ↓
修没成几次了?
   ↓ <3                  ↓ ≥3
回 Phase 1 重新分析        Phase 4.5 质疑架构
                          (停, 跟用户讨论)
```

## 危险信号

念头出现 → 立刻停下, 是偷懒借口:

| 内心戏 | 真相 |
|---|---|
| "快速改 X 试试" | "试" = 在 Phase 3 之前动手 = 违规 |
| "同时改几处一起测" | 中了不知道哪个起作用 = 没排错 |
| "跳测试我手动验证" | 手动 = 没自动化 = 下次回归不知道 |
| "大概是 X, 直接改" | "大概" = 没 Phase 1。停 |
| "已经修 3 次了再试一次" | 3+ = 架构问题, 不是假设问题 |
| "环境/玄学问题, 跳过" | 95% "玄学"是 Phase 1 没做完 |
| "stack trace 太长不想读" | 它里头有答案。读 |
| "顺手把这块也优化了" | 一次一处, "顺手"破坏验证 |

## 用户在催你时的信号

用户讲这些话 → 你在错路上, 回 Phase 1:

- "是这样吗?" — 你假设了没验证
- "能看到 X 输出吗?" — 你该加诊断没加
- "别猜了" — 你在 Phase 3 前提方案
- "深度想想" / "ultrathink" — 你在修症状不质疑根因
- "卡住了?" (用户烦躁) — 你的路不对, 停

## 何时不激活

- 纯重构 (无 bug, 改结构) — 走 plan 或直接改
- 新功能开发 — 走 plan + executing
- 预防性 hardening (无失败信号) — 不归本 skill
- subagent 失败报告 (主 agent 决定接下来怎么走)

## 衔接

- **前序**: 一般是 executing 里某 step 验证挂 → 跳这里
- **后续**: Phase 4 修完 → Phase 4 第 3 步内嵌验证 (跑命令看证据), 不交付未验证的修复
- **平行**: 多 bug 独立无共享 state → 派多个 subagent 各自走流程

## 核心原则

- **HARD-GATE 不可破** — Phase 1 没做完不准提修复
- **一次一变量** — Phase 3/4 最小改动, 多变量混改 = 没排错
- **根因 > 症状** — 在数据流源头修, 不在终点 catch
- **测试先红** — 没看到红的测试不能证明你修对了
- **3 次原则** — ≥3 次失败 = 架构问题, 不是再试一次
- **不懂说不懂** — 装懂瞎试比承认不懂代价高得多
