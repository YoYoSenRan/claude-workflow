# 测试策略

<HARD-GATE>
bug 修复必须先有能复现问题的失败验证。

行为变更必须先说明要验证的行为，再写实现。能自动化测试的，优先先写失败测试。
</HARD-GATE>

## 触发

- bug 修复
- 新功能
- 行为变更
- 重构会影响可观察行为
- 数据转换、解析器、状态机、校验、权限、计费等高风险逻辑

## 执行

1. 写出要证明的行为：
   - 正常路径
   - 关键边界
   - 错误路径
   - 回归症状（bug 修复时）
2. 能自动化时，先写最小失败测试或复现脚本。
3. 运行测试，确认失败原因对应目标行为。
4. 写最少实现让验证通过。
5. 再运行相关测试。
6. 需要整理代码时，整理后重新运行测试。

失败测试必须满足：

- 测试确实失败，而不是测试文件报错
- 失败原因对应目标行为缺失
- 不是拼写、导入或测试本身写错

## 替代验证

不能或不值得自动化测试时，用匹配任务类型的证据：

| 类型 | 验证方式 |
|---|---|
| Markdown / 文案 / workflow | 结构检查 + 对照目标逐条核对 |
| JSON / YAML / manifest | 解析或插件校验 + 关键字段核对 |
| UI | 构建/类型检查 + 浏览器或截图证据 |
| 配置 | 结构校验 + 环境差异、默认值和部署影响核对 |
| 一次性原型 | 先和用户确认是否需要自动化测试 |

替代验证格式：

```text
验证目标：
- <要证明的行为或结构>

验证方式：
- Run: <命令> — expected: <预期输出>
- 人工核对：<具体检查项>
```

## 属性测试

涉及以下类型时，建议在例子测试之外加入属性测试：

- 数据转换：转过去再转回来等于原始值
- 数学计算：通用性质保持成立
- 解析器：合法输入不崩溃，解析结果可重新序列化
- 状态机：任意操作序列不进入非法状态
- 数据校验：通过校验的数据满足所有约束

## 输出

```text
验证目标：
- ...

验证方式：
- ...

当前结果：
- ...
```

## 禁止

- 不许 bug 未复现就修。
- 不许用无意义单元测试替代文档/配置核对。
- 不许改测试来迁就实现。
- 不许测试通过后就直接声明需求完成；还要对照请求或场景。
