## 子代理辅助模式

被派遣验证测试策略时，用本 skill 只读模式。只检查测试覆盖、命令和风险；不替主智能体宣布完成或修改计划状态。

# 测试策略

test 决定怎样证明行为变化是正确的。不强制所有任务 TDD，也不替代 `verify`。

## 何时使用

补测试 / 写测试 / 设计测试策略；计划或执行前决定行为变化的测试路径；bug 根因已明确需补回归用例；重构需证明行为未变；计划中需写验证命令和预期；当前没有测试需判断替代验证是否足够。

不使用：纯文档 / README / 注释 / 小文本改动；无行为变化的格式或依赖版本改动；已是完成声明（用 `verify`）；测试失败根因未知（先 `debug`）。

## 判断顺序

1. **识别行为** — 改动影响什么用户可见行为、API、数据、边界条件。
2. **找现有测试模式** — 先读已有测试和命令，不发明新框架。
3. **选择最小测试** — 优先一条能失败再通过的回归测试。
4. **确认失败原因** — bug 修复时先看到测试因目标问题失败。
5. **交给执行流程** — 需要写代码时由 `plan` / `execute` / `debug` 执行，不在本 skill 内直接实现。
6. **跑相关验证** — 先跑目标测试，再按风险扩大范围。
7. **记录未覆盖风险** — 没覆盖的路径必须明确说明。

## 测试选择

| 场景 | 优先测试 |
|---|---|
| bug 修复 | 复现 bug 的回归测试 |
| 新功能 | 最小用户行为测试 |
| 重构 | 现有测试 + 关键路径补漏 |
| API 变更 | 契约测试、类型检查、调用方测试 |
| UI 行为 | 组件测试、交互测试或浏览器验证 |
| 配置 / 构建 | 构建、lint、启动或 dry-run |
| 文档 | 链接、命令、示例可执行性检查 |

## 例外

可以不写新测试，但必须说明原因：纯文档或说明改动；无行为变化的格式调整；测试框架不存在且本次不适合引入；成本明显高于风险的小改；只能人工或外部系统验证。

例外不等于不验证——必须给替代验证：lint、typecheck、build、截图、手工步骤或命令输出。

## 输出要求

声明用 A 风格：进行打头 `> **🔧 test** — <测试目标>`；嵌在 plan / execute 内调用时随宿主声明，不重复打头。

用户要测试策略时输出：

```text
测试目标：...

建议测试：
- 文件：...
- 场景：...
- 预期失败 / 通过条件：...

验证命令：...

不覆盖风险：...
```

计划或执行中使用本 skill 时，把测试步骤写入对应 `plan` 或 Todo，不单独漂浮在聊天里。

## 与其他 skill 的关系

| 场景 | 使用 |
|---|---|
| 需求还不清 | 回 `think` |
| 写实现计划 | `plan` 调用本 skill 的测试判断 |
| 测试失败 | `debug` |
| 完成声明 | `verify` |
| 代码评审发现缺测试 | `review` 后回 `test` 或 `plan` |
