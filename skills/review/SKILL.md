# 产出检查

> 边界：review 查"产出对需求"（做对了吗）；verify 查"声明对证据"（凭什么说做完了）。本技能负责前者。

## 触发

- 中等任务实现完成后、verify 之前
- execute 执行完所有任务后；如果要提交/PR/合并，再在 finish 之前检查
- 合并到 main 之前
- UI、API、data、config、test、docs、workflow 等任务存在质量风险时

用户给的是已收到的评审反馈时，先读 `references/receiving-feedback.md`。

## 执行

1. 读取用户原始要求、plan 或 task 场景。
2. 查看本次 diff。
3. 列出本次最容易出错的任务类型风险。
4. 逐条判断：满足 / 不满足 / 无证据。
5. 先输出问题，按严重程度排序。
6. 没有问题时，说明剩余风险和未验证项。
7. 需要独立评审时，用 `references/code-reviewer.md` 派 Agent。

## 风险表

| 任务类型 | 检查重点 |
|---|---|
| UI | 视觉状态、交互、响应式、截图或浏览器证据 |
| API | 契约、鉴权、错误码、兼容性 |
| data | 迁移、回滚、数据一致性 |
| config | 环境差异、默认值、部署影响 |
| test | 是否验证真实行为，是否覆盖失败路径 |
| docs/workflow | 文案是否会误导运行时行为，示例是否可执行 |
| refactor | 行为是否保持、调用方是否受影响 |

## 独立评审输入

使用 Claude Code 的 `Agent` 工具，填写 `references/code-reviewer.md` 中的模板。

- `{DESCRIPTION}` - 简要总结产出
- `{PLAN_OR_REQUIREMENTS}` - 它应当满足什么
- `{DIFF_COMMAND}` - 要评审的 diff 命令

DIFF_COMMAND 选择：
- 未提交改动：`git diff`
- 已暂存改动：`git diff --staged`
- 已有提交范围：`git diff <base>..<head>`

## 反馈处理

- 关键问题：立即修复
- 重要问题：继续前修复
- 次要问题：记录，除非影响本次交付
- 评审员错了：用代码、测试或需求证据回推

## 输出

```text
发现：
- [严重级别] [文件/位置] [问题] [为什么影响需求或风险]

未验证或剩余风险：
- ...
```

没有问题时：

```text
未发现需要修改的问题。
剩余风险：
- ...
```

## 禁止

- 不许先查代码质量再查需求符合度。
- 不许把测试通过当成需求满足。
- 不许盲从评审员；反馈必须能被代码、测试或需求支持。
- 不许简单低风险任务强行升级成独立评审。
