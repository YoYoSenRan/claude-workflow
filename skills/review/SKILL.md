<SUBAGENT-STOP>
如果你是被派来执行具体实现任务的子代理，请跳过此 skill。除非任务明确要求 code review，否则不要自行转评审。

# 代码评审

review 只做两类事：评审代码，或处理已经收到的评审反馈。它不负责实现新功能、排查未知根因或收尾提交。

<HARD-GATE>
评审输出必须 Findings 先行。不要先写总结、夸赞、改动概述。

没有证据的意见不能当问题。每个问题都必须指向具体文件、行号、行为风险或测试缺口。
</HARD-GATE>

## 何时使用

使用本 skill：

- 用户说“review / 帮我审一下 / 看看这个 diff / 检查 PR”；
- 用户贴了代码评审反馈，要求判断或处理；
- 用户问某个改动有没有风险；
- 合并前需要从 bug、回归、安全和测试角度审一遍。

不使用本 skill：

- bug 根因不明，应该用 `debug`；
- 用户要求按计划实现，应该用 `execute`；
- 完成前确认是否可交付，应该用 `verify`；
- 收尾提交或 PR，应该用 `finish`。

---

## 评审流程

按顺序完成：

1. **确认评审范围** — diff、文件、分支、PR 或用户贴出的反馈。
2. **读取上下文** — 至少读取相关 diff 和被影响文件；不能只看文件名。
3. **按风险审查** — bug、行为回归、安全、数据损坏、缺失测试优先。
4. **写 Findings** — 有问题就按严重性排序。
5. **列开放问题** — 只有确实影响判断时才列。
6. **补充摘要** — 摘要放最后，且保持简短。

如果没有发现问题，直接说没有发现阻塞问题，并说明未覆盖的风险。

---

## Findings 格式

每个 finding 使用：

```text
- [严重性] 文件:行号 — 问题标题
  具体说明为什么这是 bug、回归、安全风险或测试缺口；说明触发条件和影响。
```

严重性：

- `高`：会导致错误行为、数据损坏、安全问题、构建/测试失败。
- `中`：明显回归、边界条件失败、缺失关键测试。
- `低`：可维护性风险或小范围行为不一致。

规则：

- 不写没有证据的“可能有问题”。
- 不把风格偏好包装成 bug。
- 不因命名、格式、主观偏好给阻塞意见。
- 同类问题合并，避免重复刷屏。

---

## 接收评审反馈

当用户贴出别人给的 review 意见时：

1. 先判断反馈是否成立；
2. 成立则给最小修复建议；
3. 不成立则说明原因和证据；
4. 反馈含糊时要求补上下文，不盲改；
5. 如果要执行修改，转 `think` / `execute` / `debug`，不要在 `review` 内擅自实现。

---

## 输出结构

有问题：

```text
Findings
- [高] path/to/file.ts:42 — 标题
  说明...

开放问题
- ...

摘要
简短说明审查范围。
```

无问题：

```text
Findings
未发现阻塞问题。

剩余风险
- 未运行测试 / 未覆盖某路径 / 只审查了指定 diff。
```

## 警示信号

| 念头 | 现实 |
|---|---|
| "先总结一下改了什么" | review 要 Findings 先行。 |
| "这里写法我不喜欢" | 偏好不是 bug。 |
| "感觉可能有问题" | 没有证据就不要列 finding。 |
| "反馈看起来对，直接改吧" | 先验证反馈是否成立。 |
| "没问题就夸一下" | 没问题时说清剩余风险即可。 |
