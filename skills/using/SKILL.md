# 工作流技能
Claude Workflow 是一组协同工作的开发流程 skills，不是零散命令集合。本入口负责分诊：收到请求先判信号、再选能安全完成任务的最小流程，然后路由到对应 skill。其他 skill 的正文按需加载。
选最小足够流程：小任务不流程化，模糊任务不盲改，高风险任务不跳过计划和验证。某 skill 明确适用时必须调用并遵守，不因“可能相关”就把低风险小改升级成完整流程。

## 路由三问

收到请求按顺序判，命中即停、即路由：

1. **要改文件吗？** 否 → 简单事实直接答；只读分析或方案判断用 `think`。
2. **目标和范围清楚吗？** 否 → 用 `think` 复述理解、提关键问题对齐，禁止盲改。
3. **有失败信号、且要你修复吗？**（报错 / 测试挂 / 构建失败 / 行为异常）是 → 用 `debug` 找根因，禁止症状级补丁；只要求解释报错或代码含义、不修 → `think`。

三问都过（要改、清楚、无失败信号）→ 按下方表选改动粒度。

## 入口路由

| 请求类型 | 使用方式 |
|---|---|
| 简单事实 / 单行命令 | 直接回答或运行命令 |
| 清晰小改 | 读目标文件 → 最小修改 → 轻量验证 |
| 只读分析 / 方案判断 / 模糊实现 | 使用 `think` |
| 复杂实现 / 多步骤改动 | `think` → `plan` |
| 执行已有计划 | 使用 `execute` |
| bug / 报错 / 测试失败 | 使用 `debug` |
| 测试策略 / 补回归测试 | 使用 `test` |
| 完成声明 | 使用 `verify` |
| 收尾 / 提交 / PR | 使用 `finish` |
| 撤销 / 回退 / 丢弃改动 | 使用 `finish`（破坏性，必须二次确认） |
| 代码评审 | 使用 `review` |
| 隔离工作区 | 使用 `worktree` |
| 子代理调度 | 使用 `subagent` |
| 创建 / 修改 workflow skill | 使用 `skill` |
| 初始化项目 rules / skills / references | 使用 `setup` |

消歧：显式要求派子代理（含派子代理做评审）先走 `subagent`，不直接进 `review`；需求未确认就要写计划时先回 `think`。

轻量小改（明确目标、只动单文件或少量文本、低风险、验证简单）直接最小修改完成；涉及多文件、公共 API、架构、迁移、删除、重命名、批量替换，路由到 `think` 判定。完整判定标准见 `think`。

## 兜底

拿不准，或一个请求同时命中多类时，往“只读 / 对齐”一侧倒：先用 `think` 看清再动，不默认动手。宁可多问一句，不要错改一处。

涉及撤销、回退、丢弃、重置、删除等不可逆操作时，先确认具体范围再执行，绝不默认动手——绝不在未确认时擅自 `git checkout` / `reset` / `restore` 覆盖用户改动。

## 指令优先级

用户显式指令（AGENTS.md、CLAUDE.md、直接请求）> Claude Workflow skills > 默认系统提示。用户定义做什么，skill 定义怎么做。冲突时遵循用户指令，并简短说明被跳过的流程风险。

## 项目级 rules 和 skills

- 存在 `CLAUDE.md`、`.claude/CLAUDE.md` 或 `.claude/rules/` 时，它们是项目级持续规则，优先于通用 workflow 默认建议。
- 存在 setup 生成的 `.claude/skills/` 时，先按实际存在的 `.claude/skills/*/SKILL.md` 名称匹配当前任务，只加载最相关的一个。
- 找不到匹配，改读适用的 `.claude/rules/`；reference 由对应 skill 自己 Read。
- 项目画像、命令矩阵、扫描报告按 reference 处理；skill 按项目真实任务能力命名。

## 调用纪律

- 某 skill 明确适用，先用 `Skill` 工具加载并按原文执行：不凭记忆，不用普通文件读取代替。
- 调用任何产出用户交付物的 skill 时用统一 A 风格声明，进行打头、完成收口：

  ```text
  > **🔧 <skill>** — <目的>
  ...
  > **✅ 完成** — 验证 `<命令>` · 下一步 `<skill>`
  ```

  轻量小改压成单行 `> **🔧** <做什么> · ✅ <验证>`；只读分析不强制完成收口。
- 非轻量任务有清单就用 TodoWrite。
- 子代理只执行被派遣的具体任务，不递归加载完整主流程。

## 警示信号

| 念头 | 现实 |
|---|---|
| "先套完整流程再说" | 错。先判路由三问，选最小足够流程。 |
| "模糊需求可以直接做" | 错。目标或范围不清时用 `think` 对齐。 |
| "失败了先试个补丁" | 错。有失败信号时用 `debug` 找根因。 |
| "我记得这个 skill" | skill 会变，必须加载当前版本。 |
