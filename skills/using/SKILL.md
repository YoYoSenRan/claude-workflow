## 指令优先级

技能会覆盖默认系统提示词的行为，但**用户的指令始终优先**：

1. **用户的显式指令**（CLAUDE.md、GEMINI.md、AGENTS.md、直接请求）—— 最高优先级
2. **技能** —— 在与默认系统行为冲突时覆盖之
3. **默认系统提示词** —— 最低优先级

如果 CLAUDE.md、GEMINI.md 或 AGENTS.md 写着"不要使用 TDD"，而某个技能说"始终使用 TDD"，请遵循用户的指令。用户拥有控制权。

## 如何访问技能

**在 Claude Code 中：** 使用 `Skill` 工具。当你调用一个技能时，其内容会被加载并呈现给你——直接遵循它即可。永远不要用 Read 工具去读技能文件。

**在其他环境中：** 请查阅你所用平台的文档，了解技能是如何加载的。

# 使用技能

## 规则

**在任何回复或行动之前，先调用相关或被请求的技能。** 即便只有 1% 的可能某个技能适用，你也应当调用该技能去检查。如果调用后发现该技能不适合当前情形，你不必使用它。

### 执行步骤

收到用户消息（包括澄清提问）时：

1. **是否即将进入 PlanMode？**
   - 是 + 本会话未做过 think → 先调用 think 技能
   - 其他情况 → 跳到步骤 2
2. **是否有任何技能 ≥1% 可能适用？**
   - 否 → 直接回复（包括澄清提问）
   - 是 → 进入步骤 3
3. 调用 `Skill` 工具
4. **公开宣告**：`正在使用 [技能名] 来 [用途]`
5. **技能内是否有 checklist？**
   - 是 → 用 TodoWrite 为每一项建立 todo
   - 否 → 进入步骤 6
6. **严格按技能执行**——不裁剪、不二次解释、不"我觉得这条可以跳过"

### 工作流入口违规对照

| 违规念头 | 实际在做 | 必须改为 |
|---|---|---|
| "这次太简单不需要技能" | 跳过 1% 原则 | 调用技能 |
| "先回复再说，回头补技能" | 偷偷绕过入口 | 先调技能再回复 |
| "用了技能不必宣告" | 违反明牌义务 | 必须公开宣告 |
| "checklist 我记得住" | 用脑子代替工具 | TodoWrite 落地每一项 |
| "这条不太合适，跳过" | 二次解释技能 | 严格执行原文 |
| "只是个澄清问题" | 用"小回复"豁免规则 | 澄清前也要扫技能 |
| "我已经 brainstorm 过类似的了" | PlanMode 前置门失守 | 本会话未做过就要做 |

### 正反例

**正例**

```
用户："帮我重构这个函数"
你：正在使用 think 技能来在改代码前厘清范围。
    [调用 Skill(think)]
    [按技能步骤执行]
```

```
用户："今天巴黎天气怎么样"
你：[扫描技能列表 → 无相关技能]
    [直接回复天气]
```

**反例**（不要这样做）

```
用户："帮我重构这个函数"
你："好的，我先看下代码……"
                            ↑ 错：跳过技能扫描，直接动手
```

```
用户："进入 plan mode 写个登录功能"
你：[直接 EnterPlanMode]
                            ↑ 错：未先调用 think
```

```
用户："加一个测试"
你："我来写测试……"
    [调用 test-driven-development 技能但未宣告]
                            ↑ 错：违反明牌义务（应先说"正在使用 test-driven-development 技能来加测试"）
```

### 边界澄清

- **澄清提问也算回复** → 提问前同样要扫技能
- **空消息 / 纯打招呼**（"在吗"）→ 无技能适用，直接回复
- **多个技能同时适用** → 全部调用，按相关度从高到低宣告
- **不确定哪个技能适用** → 宁滥勿缺，1% 原则兜底

## 危险信号

以下念头意味着停下——你正在自我合理化：

| 念头 | 现实 |
|---------|---------|
| "这只是个简单问题" | 提问就是任务。检查是否有技能。 |
| "我需要先获取更多上下文" | 检查技能要在澄清提问之前。 |
| "让我先探索一下代码库" | 技能会告诉你如何探索。先检查。 |
| "我可以快速看一下 git/文件" | 文件缺少对话上下文。检查是否有技能。 |
| "我先收集一下信息" | 技能会告诉你如何收集信息。 |
| "这不需要一个正式的技能" | 如果存在技能，就使用它。 |
| "我记得这个技能" | 技能会演进。读当前版本。 |
| "这不算一个任务" | 行动即任务。检查是否有技能。 |
| "用这个技能太过了" | 简单的事情会变复杂。用它。 |
| "我就先做这一件事" | 在做任何事之前先检查。 |
| "这感觉很有效率" | 无纪律的行动浪费时间。技能避免这一点。 |
| "我知道那是什么意思" | 知道概念 ≠ 使用技能。调用它。 |

## 技能优先级

当多个技能都可能适用时，按以下顺序使用：

1. **先用流程类技能**（think、debugging）——这些决定**如何**处理任务
2. **再用实现类技能**（frontend-design、mcp-builder）——这些指导执行

"我们来构建 X" → 先 think，然后是实现类技能。
"修复这个 bug" → 先 debugging，然后是领域相关的技能。

## 技能类型

**刚性**（TDD、debugging）：严格遵循。不要把纪律性变通掉。

**柔性**（模式类）：将原则适配到上下文。

技能本身会告诉你属于哪一类。

## 用户指令

指令说的是"做什么"，不是"怎么做"。"添加 X"或"修复 Y"并不意味着跳过工作流。
