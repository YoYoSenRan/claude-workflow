# 技能入口

先判断用户要做什么，再交给一个主流程。using 只做入口路由，不展开下游步骤。

## 执行

1. 在回复、读文件、跑命令或编辑前，先判断是否有 skill 适用。
2. 有适用 skill 时，调用 Skill 工具加载当前版本。
3. 多个入口都像适用时，只选最贴近用户意图的一个。
4. 主流程没要求时，不主动加载 plan、execute、subagent、test、verify、finish。

本入口每轮都适用，多轮对话、上下文压缩后不失效；不确定时仍按它判断。

## 路由

先分清用户要"答案"还是要"动手"。只要答案、不依赖改项目代码的解释或闲聊，直接回答，不入主流程。需要读项目代码再做判断或评价的，仍走 think 的只读分析。

| 用户意图 | 入口 |
|---|---|
| 新功能、新组件、改行为、重构、架构判断、代码理解、设计评价 | think |
| bug、报错、测试失败、异常行为 | debug |
| 检查产出、评审改动、合并前把关 | review |
| 初始化或更新项目工作流 | setup |
| 创建或编辑技能 | skill |

## 主从关系

- `think/debug/setup/review/skill` 是主流程入口。
- `plan/execute/subagent/test/verify/finish` 是下游能力。
- `plan` 写完后等用户确认，除非用户已经明确说继续执行。

## 输出

对用户只说你正在做的任务动作。除非用户正在讨论 workflow 本身，不提 skill 名称、不说"加载"。

## 禁止

- 不许凭记忆执行 skill；必须加载当前版本。
- 不许让多个主流程同时控制同一任务。
- 不许把下游能力当成入口。
- 不许把纯解释、问答或闲聊硬塞进主流程。
