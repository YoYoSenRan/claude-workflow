## 子代理辅助模式

被派遣做方案调研、上下文整理或风险分析时，用本 skill 的分析模式。只返回发现、判断、风险和建议；不替主智能体确认需求、写计划、改文件或宣布方向已批准。

# 思考与对齐

把粗略请求变成明确判断、设计方向或可执行任务。核心不是多问，是避免在不了解上下文时乱改。

<HARD-GATE>
用户没有要求实现或修改时，禁止编辑文件。

实现类请求如果范围、目标或验收标准不清晰，禁止直接改代码；必须先复述理解、提出关键问题或给出推荐方案并等待确认。
</HARD-GATE>

## 清单

1. **识别模式** — 分析模式还是实现意图模式。
2. **确认边界** — 本轮是只读分析、写计划，还是允许改文件。
3. **收集上下文** — 分析模式可广读；实现模式只读与目标直接相关的文件。
4. **给出判断或对齐** — 分析给结论；模糊实现先问或给推荐方案。
5. **判复杂度并路由** — 见下方路由表。
6. **等待确认** — 需要实现、架构调整或多文件计划时，等用户确认再进入下一步。

## 分析模式

触发见 description 和下方模糊信号表。

允许：读项目文件、文档、git 历史、参考项目、官方资料；输出当前状态、主要问题、推荐方向、风险和下一步。

禁止：未经要求编辑文件；把分析强行变成实现计划；无证据下结论。

## 实现意图模式

### 轻量小改

本节是轻量小改判定的唯一来源；`using` 入口只做一句话粗判，细则不在 `using` 复制。

满足全部条件可直接执行：

- 用户给出明确文件、函数、行号或具体行为；
- 只改 1 个文件或少量文本；
- 不影响公共 API、架构、数据迁移、删除、重命名或批量替换；
- 验证方式简单。

流程：读目标文件 → 只做指定改动 → 最小验证（格式 / 静态检查 / 相关命令 / 人工核对）→ 简短汇报。

禁止：写规格说明、写 plan、扩大范围、顺手重构。

### 模糊实现

先输出：

```text
我理解你想做的是：<具体动作>，范围可能是 <A/B>。这里有两种解读：...
我建议先按 <推荐项> 做，因为 <理由>。方向对不对？
```

必要时只问 1 个最关键问题，不要一次问一堆。

### 复杂实现

满足任一条件则不要直接改：

- 跨多个文件或模块；
- 涉及架构、公共 API、数据迁移；
- 需要新增工作流、测试体系或同步机制；
- 取舍还需用户确认。

处理：先写 `docs/specs/YYYY-MM-DD-<topic>.md`（只记意图、范围、约束、不做什么），再转 `plan`。

## 模糊信号

命中这些信号时，不要直接改文件：

| 信号 | 例子 | 处理 |
|---|---|---|
| 指代不明 | "这个"、"这块"、"那个模块" | 问范围或先只读定位 |
| 目标不明 | "优化一下"、"处理一下" | 问目标或给推荐方案 |
| 多解读 | 既可能是分析，也可能是实现 | 先复述当前理解 |
| 高代价 | 删除、迁移、重命名、架构调整 | 先给方案和风险 |
| 多文件 | 跨模块、跨流程、公共 API | 转规格说明 / plan |
| 连续确认 | "好"、"继续"、"下一步" | 只代表继续上一阶段；如果上一阶段只是分析或建议，不能当作执行批准 |

## 路由

| 任务类型 | 下一步 |
|---|---|
| 只读分析 / 方案判断 | 本 skill 内完成，给结论 |
| 轻量小改 | 读取目标文件后直接执行，不写规格说明 / plan |
| 多步骤实现 / 多文件改动 | 写规格说明，然后转 `plan` |
| 已有明确计划并要求执行 | 转 `execute` |
| bug / 测试失败 / 报错 / 异常行为 | 转 `debug` |
| 完成前确认是否可交付 | 转 `verify` |

## 规格说明规则

复杂实现需要规格说明时，路径为 `docs/specs/YYYY-MM-DD-<topic>.md`。

只写：

```markdown
# <topic>

**目标：** 一句话目的
**范围：** 改什么 / 不改什么
**约束：** 兼容性、性能、风格、工具约束
**不做：** 明确不做什么
**待确认问题：** 计划阶段可解决的问题
```

不写具体代码、任务拆解、验证命令、TDD 步骤——这些属于 `plan`。

## 警示信号

| 念头 | 现实 |
|---|---|
| "先改了再说" | 模糊实现必须先对齐。 |
| "分析也不能读文件" | 错。分析模式必须读上下文。 |
| "用户说看看，那我先问一堆问题" | 错。项目分析应先只读了解现状。 |
| "这次跨很多文件但我能直接做" | 跨文件通常需要规格说明 / plan。 |
| "我给三个方案就够了" | 还要给推荐和理由。 |
| "他让继续，说明所有细节都确认了" | 只代表方向确认，执行仍要按边界推进。 |

## 何时不激活

- 用户只问事实，且答案不依赖项目上下文。
- 用户给出明确单行命令，比如 `git status`、`date`。
- 子代理任务已由主智能体拆好。
- 正在严格执行已批准 plan，应该用 `execute`。
