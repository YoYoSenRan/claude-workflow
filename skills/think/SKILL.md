# 动手之前先想清楚

<HARD-GATE>
在用户确认方案之前，不得写任何代码、不得搭建任何项目脚手架、不得采取任何实现动作。
</HARD-GATE>

think 是整个工作流的入口。它深度分析用户的需求，理解涉及的范围和风险，然后给出完整的执行路径——包括后续会走哪些技能、什么顺序、怎么验证。分析深度匹配任务复杂度。

## 何时使用

创建新功能、构建组件、新增能力或修改行为之前。bug 修复走 debug 技能，不走 think。

## 流程

### 第 1 步：理解需求

- 快速看一下相关文件和上下文
- 需求不清楚？问**一个**澄清问题（优先选择题）
- 需求涉及多个不相关的子系统？立刻指出来，建议拆分
- 关注三件事：**目的、约束、成功标准**

### 第 2 步：评估复杂度

根据分析结果判断任务属于哪一级：

**简单** — 改什么、怎么改已经很明显
- 例：加个配置项、改个组件的 props、调整一处逻辑
- 涉及 1-2 个文件，无设计决策

**中等** — 目标清楚，有一些设计选择但能快速说清
- 例：加一个 API 端点、实现一个新组件、改一个模块的行为
- 能在一个会话内完成

**复杂** — 多个任务、跨子系统、需要写下来追踪进度
- 例：重构认证系统、做一个完整功能（前后端 + 数据库）、架构级变更
- 可能跨会话、需要子代理分工、或需要别人接手

### 第 3 步：规划完整的执行路径

分析完复杂度后，向用户呈现**完整的执行方案**——不只是"下一步做什么"，而是从现在到完成的整条路径。

#### 简单任务的路径

```
think 确认方案 → 实现（TDD）→ verify 验证 → 完成
```

向用户呈现：
- 改哪个文件、改什么
- 怎么验证（跑什么命令）

示例：

> "我在 `src/config.ts` 加一个 `maxRetries` 配置项，默认值 3。验证：跑 `npm test`。确认？"

不需要 worktree、不需要 plan、不需要 review。确认后直接实现。

#### 中等任务的路径

```
think 确认方案 → [worktree 如果需要隔离] → 实现（TDD）→ review 评审 → verify 验证 → finish 收尾
```

向用户呈现：
- 方案概要（涉及的文件、设计选择、验证方式）
- 是否需要隔离工作区
- 完成后走 review 还是直接 verify

示例：

> "方案：在 `src/api/users.ts` 加 `DELETE /users/:id` 端点，在 `tests/api/users.test.ts` 加对应测试，在 `src/middleware/auth.ts` 加权限检查。
>
> 不需要隔离工作区（改动可控）。按 TDD 实现，完成后跑 `npm test` 验证。确认？"

#### 复杂任务的路径

```
think 确认方向 → plan 写设计+方案 → worktree 创建隔离工作区 → subagent/execute 执行 → review 评审 → verify 验证 → finish 收尾
```

向用户呈现：
- 对任务的整体判断（为什么是复杂任务）
- 涉及的子系统和大致范围
- 建议的执行方式（子代理 or 内联）

示例：

> "这涉及认证、会话管理和数据库迁移三个子系统，预计 5 个子任务。建议：
> 1. 先写正式方案（plan）
> 2. 建隔离工作区
> 3. 用子代理逐任务执行，每任务后评审
> 4. 全部完成后走 finish 收尾
>
> 我先写方案？"

### 第 4 步：用户确认

等用户确认后再行动：
- 简单/中等：确认后直接开始实现
- 复杂：确认后调用 plan 技能

用户可能调整方案（"不需要 worktree"、"这个不用 review"、"我觉得不需要 plan 直接做"），遵循用户的判断。

### 在已有代码库中工作

- 提出改动前先看当前结构，遵循已有模式
- 不要提议无关的重构，聚焦当前目标
- 当本次触碰的代码有实际缺陷或会影响交付的边界问题，把最小修正纳入方案

## 工作流全景

think 需要知道所有下游技能及其触发条件：

| 技能 | 何时使用 | 谁触发 |
|---|---|---|
| **plan** | 复杂任务需要写方案文件 | think 路由 |
| **worktree** | 需要隔离当前工作区时 | think 建议，实现前执行 |
| **test** | 所有实现都要遵循 TDD | 实现过程中自动遵循 |
| **execute** | 按方案文件逐步执行（内联） | plan 交接 |
| **subagent** | 按方案文件派发子代理执行 | plan 交接 |
| **review** | 中等/复杂任务完成后 | 实现完成后 |
| **debug** | 实现过程中遇到 bug | 随时 |
| **verify** | 宣称完成之前 | finish 之前 |
| **finish** | 决定如何整合工作成果 | verify 之后 |

<constraints>
- 禁止在用户确认方案前写代码
- 禁止需求不清楚时猜测——先问
- 禁止提议无关的重构
- 禁止把所有任务都当复杂任务处理——匹配复杂度
- 禁止只说"下一步"而不呈现完整路径
</constraints>

## 警示信号

| 念头 | 现实 |
|---|---|
| "这太简单了不需要想" | 简单也要确认一句，不是跳过 |
| "让我先写着看" | 先确认再写 |
| "这个需要写完整设计文档" | 先评估复杂度——可能几句话就够了 |
| "所有任务都走 plan" | 只有复杂任务才需要 plan |
| "确认了方案就开始写代码" | 复杂任务确认的是方向，还要等 plan 批准 |
| "用户说做就做，不用分析" | think 的分析是为了做对，不是为了拖延 |
