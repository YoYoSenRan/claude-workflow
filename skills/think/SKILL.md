## 子代理辅助模式

如果你是作为子代理被派遣做方案调研、上下文整理或风险分析，可以使用本 skill 的分析模式。
只返回发现、判断、风险和建议；不要替主智能体确认需求、写计划、改文件或宣布方向已批准。

# 思考与对齐

把粗略请求变成明确判断、设计方向或可执行任务。核心不是多问，而是避免在不了解上下文时乱改。

<HARD-GATE>
用户没有要求实现或修改时，禁止编辑文件。

实现类请求如果范围、目标或验收标准不清晰，禁止直接改代码；必须先复述理解、提出关键问题或给出推荐方案并等待确认。
</HARD-GATE>

## 两种模式

### 1. 分析模式

触发：

- "分析一下当前项目"
- "看看这个项目有没有问题"
- "这个设计合理吗"
- "帮我对比 X 和 Y"
- "为什么现在做得不好"

允许：

- 读取项目文件、文档、git 历史和参考项目。
- 查官方文档或行业参考。
- 输出判断、问题、风险和建议。

禁止：

- 未经用户要求编辑文件。
- 把分析任务强行变成实现计划。
- 在没有证据时给结论。

### 2. 实现意图模式

触发：

- "帮我实现"，但目标或边界还不清楚
- "改一下"，但没有明确文件、行为或完成标准
- "优化一下"，但没有说明优化目标
- "重构这块"，但范围、风险或验收方式不清楚
- "按这个方案做"

规则：

- 请求清晰：先读取目标文件，再做最小范围改动。
- 请求模糊：先复述、提问或给推荐方案，等确认。
- 多文件或架构级改动：先写 spec 或转 `plan`，不直接开改。
- 明确单文件小改：可以不写 spec / plan，读取目标文件后直接最小修改。

---

## 清单

按顺序完成：

1. **识别模式** — 判断是分析模式还是实现意图模式。
2. **确认边界** — 明确本轮是只读分析、写计划，还是允许改文件。
3. **收集上下文** — 分析模式可先读文件；实现模式只读与目标直接相关的文件。
4. **给出判断或对齐问题** — 分析给结论；模糊实现先问或给推荐方案。
5. **判复杂度** — 决定直接小改、写 spec、转 `plan`、转 `debug` 或转 `execute`。
6. **等待确认** — 需要实现、架构调整或多文件计划时，等用户确认后再进入下一步。

## 模糊信号

命中这些信号时，不要直接改文件：

| 信号 | 例子 | 处理 |
|---|---|---|
| 指代不明 | "这个"、"这块"、"那个模块" | 问范围或先只读定位 |
| 目标不明 | "优化一下"、"处理一下" | 问目标或给推荐方案 |
| 多解读 | 既可能是分析，也可能是实现 | 先复述当前理解 |
| 高代价 | 删除、迁移、重命名、架构调整 | 先给方案和风险 |
| 多文件 | 跨模块、跨流程、公共 API | 转 spec / plan |
| 连续确认 | "好"、"继续"、"下一步" | 只代表继续上一阶段；如果上一阶段只是分析或建议，不能当作执行批准 |

## 分析模式流程

1. 说明本轮是只读分析。
2. 读取必要上下文：
   - 项目结构；
   - 相关文档；
   - 关键源码；
   - 最近 git 状态或提交；
   - 用户指定参考项目或官方资料。
3. 输出：
   - 当前状态；
   - 主要问题；
   - 推荐方向；
   - 风险和下一步。
4. 不编辑文件，除非用户明确说继续实施。

---

## 实现意图模式流程

### 轻量小改

满足以下条件可直接执行：

- 用户给出明确文件、函数、行号或具体行为；
- 预计只改 1 个文件或少量文本；
- 不影响公共 API、架构、数据迁移、删除、重命名或批量替换；
- 验证方式简单。

流程：

1. 读取目标文件。
2. 只做用户指定改动。
3. 做最小验证：格式检查、静态检查、相关命令或人工核对。
4. 简短汇报改了什么和如何验证。

禁止：

- 写 spec。
- 写 plan。
- 扩大范围。
- 顺手重构。

### 模糊实现

先输出：

```text
我理解你想做的是：<具体动作>，范围可能是 <A/B>。这里有两种解读：...
我建议先按 <推荐项> 做，因为 <理由>。方向对不对？
```

必要时只问 1 个最关键问题。不要一次问一堆开放题。

### 复杂实现

满足任一条件则不要直接改：

- 跨多个文件或模块；
- 涉及架构、公共 API、数据迁移；
- 需要新增工作流、测试体系或同步机制；
- 需求还需要用户确认取舍。

处理方式：

- 先写 `docs/specs/YYYY-MM-DD-<topic>.md`，只记录意图、范围、约束和不做什么；
- 再转 `plan` 写可执行计划。

## 路由

| 任务类型 | 下一步 |
|---|---|
| 只读分析 / 方案判断 | 本 skill 内完成，给结论 |
| 轻量小改 | 读取目标文件后直接执行，不写 spec / plan |
| 多步骤实现 / 多文件改动 | 写 spec，然后转 `plan` |
| 已有明确计划并要求执行 | 转 `execute` |
| bug / 测试失败 / 报错 / 异常行为 | 转 `debug` |
| 完成前确认是否可交付 | 转 `verify` |

## spec 规则

复杂实现任务需要 spec 时，路径为：

```text
docs/specs/YYYY-MM-DD-<topic>.md
```

spec 只写：

```markdown
# <topic>

**目标：** 一句话目的
**范围：** 改什么 / 不改什么
**约束：** 兼容性、性能、风格、工具约束
**不做：** 明确不做什么
**待确认问题：** 计划阶段可解决的问题
```

spec 不写：

- 具体代码；
- 任务拆解；
- 验证命令；
- TDD 步骤。

这些属于 `plan`。

---

## 关键原则

- 先分清是分析还是实现。
- 分析可以读上下文；实现不能盲改。
- 推荐方案要有立场，不只堆选项。
- 简单小改不要过度文档化。
- 复杂改动不要跳过 spec / plan。
- 用户确认前，不把方案当批准。

## 警示信号

| 念头 | 现实 |
|---|---|
| "先改了再说" | 模糊实现必须先对齐。 |
| "分析也不能读文件" | 错。分析模式必须读上下文。 |
| "用户说看看，那我先问一堆问题" | 错。项目分析应先只读了解现状。 |
| "这次跨很多文件但我能直接做" | 跨文件通常需要 spec / plan。 |
| "我给三个方案就够了" | 还要给推荐和理由。 |
| "他让继续，说明所有细节都确认了" | 只代表方向确认，执行仍要按边界推进。 |

## 何时不激活

- 用户只是问事实，且答案不依赖项目上下文。
- 用户给出明确单行命令，比如 `git status`、`date`。
- 子代理任务已经由主智能体拆好。
- 当前正在严格执行已批准 plan，应该使用 `execute`。
