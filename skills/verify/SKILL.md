## 子代理辅助模式

被派遣做验证时，运行或检查指定验证项，只报告命令、退出状态、关键输出和覆盖范围；不替主智能体做最终完成、可交付或合并声明。

# 完成前验证

verify 是完成声明前的证据门。只回答“现在是否有足够证据说完成”，不负责继续实现、提交、PR 或评审。

<HARD-GATE>
没有新鲜验证证据前，禁止说“完成 / 修好 / 通过 / 可用 / 可以交付”。

新鲜证据必须来自当前任务上下文中真实运行的命令、检查或人工可核对结果；历史输出、推测、计划里的预期、子代理转述都不能单独作为完成证据。

如果同一轮刚运行过最终验证，并且命令、退出状态和关键输出仍可引用，可以复用；跨轮、compact 后、输出不可见或声明对象变化时，必须重新验证。
</HARD-GATE>

## 何时使用

用户问“完成了吗 / 可以交付了吗 / 修好了吗 / 测试过了吗”；`execute` 完成全部任务后准备汇报；`debug` 修复后准备声明已修复；任何回复准备声称通过 / 可用 / 完成 / 可交付。

不使用：还没开始实现（用 `think` / `plan`）；正在按计划执行（继续 `execute`）；bug 根因不清（继续 `debug`）；用户只要 code review（用 `review`）。

## 清单

1. **明确声明对象** — 要证明的是功能完成、bug 修好、测试通过，还是计划任务完成。
2. **选择验证命令** — 按下方验证选择优先级。
3. **运行新鲜验证** — 真实执行；只有同一轮可引用的最终验证输出可复用。
4. **读取完整输出** — 关注退出码、失败片段、跳过项和警告。
5. **判断证据是否足够** — 不足就说不足，不包装成完成。
6. **汇报结论** — 必须包含验证命令和结果。

## 验证选择

优先级：① 计划文件 `## 验证方式` 列出的最终验证；② 当前任务或 bug 直接相关的测试；③ 类型检查、lint、build 等项目级质量门；④ 文档 / 配置 / 纯说明变更的格式检查或人工核对项。

没有合适命令时必须说明：

```text
目前没有足够的自动化验证命令能证明完成。
我已核对：<人工检查项>。
剩余风险：<无法自动验证的部分>。
```

不要为了看起来完整而运行无关命令。

## 结论格式

声明用 A 风格：进行打头 `> **🔧 verify** — <声明对象>`，结论即收口（✅ 通过 / ❌ 失败 / ⚠️ 证据不足）。

通过：

```text
可以说完成。新鲜验证：
- `<命令>`：通过，关键输出 <摘要>
```

失败：

```text
现在不能说完成。验证失败：
- `<命令>`：失败，关键输出 <摘要>
下一步应回到 <execute/debug/plan> 处理。
```

证据不足：

```text
现在不能严格说完成。原因：缺少能证明 <声明对象> 的新鲜验证。
已完成的检查：<检查项>
剩余风险：<风险>
```

## 警示信号

| 念头 | 现实 |
|---|---|
| "刚才已经跑过了" | 只有同一轮可引用的命令、退出状态和关键输出才算证据。 |
| "计划里写了预期会通过" | 预期不是证据。 |
| "子代理说通过了" | 需要主智能体读取或复核关键输出。 |
| "只是文档不用验证" | 也要做格式、路径或人工核对。 |
| "大概率没问题" | 大概率不能支撑完成声明。 |

## 何时停止

验证命令失败；命令不存在或无法运行；输出和预期不一致；验证范围不能覆盖声明对象。停止时报告具体命令、输出摘要和下一步应回到哪个 skill。
