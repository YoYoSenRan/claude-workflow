# 完成前的验证

<HARD-GATE>
没有新鲜证据，不得声明完成、修好、通过或满足需求。
</HARD-GATE>

> 边界：verify 查"声明对证据"（凭什么说做完了）；review 查"产出对需求"（做对了吗）。本技能负责前者。

## 触发

在以下动作之前使用：

- 成功、完成、修好、通过、满足需求等声明
- 提交、创建 PR、标记任务完成
- 切换到下一项任务或委派给代理

## 执行

1. 写清要声明什么。
2. 重读用户请求、plan 或 task 场景。
3. 为每个声明选择证据：命令、截图、人工核对或 diff 检查。
4. 重新运行命令或重新执行检查。
5. 读取完整输出，确认退出码、失败数和关键结果。
6. 对照声明逐项判断：证据充分 / 不充分。
7. 只声明证据支持的内容。

## 证据表

| 声明 | 需要的证据 | 不足以证明 |
|-------|----------|----------------|
| 测试通过 | 测试命令输出：0 失败 | 上一次运行结果、"应该会过" |
| Linter 干净 | Linter 输出：0 错误 | 局部检查、外推 |
| 构建成功 | 构建命令：exit 0 | Linter 通过、日志看上去不错 |
| Bug 修好 | 测试原始症状：通过 | 改了代码、自以为修好 |
| 没弄坏别的东西 | 相关回归验证通过；高风险修复需红-绿核验 | 测试通过一次 |
| 代理已完成 | VCS diff 显示有改动 | 代理自我汇报"成功" |
| 满足需求 | 逐条对照清单 | 测试通过 |
| 命令通过 | 明确 exit 0，且输出未超时/未截断 | `timeout`、被截断输出、只看最后几行 |
| UI 完成 | 构建/类型检查 + 实际页面检查或说明未做 | 只跑 typecheck 就说视觉完成 |
| 文档/规则完成 | 对照目标逐条检查，确认没有误导运行时 | Markdown 格式看起来正常 |
| 配置完成 | 结构校验 + 关键环境/默认值检查 | 只看 JSON/YAML 能解析 |

## 输出

```text
验证：
- `<命令或检查>`：通过/失败，关键结果

结论：
- 已被证据支持：...
- 未验证或仍有风险：...
```

## 禁止

- 不许用"应该"、"大概"、"似乎"描述验证结果。
- 不许用超时、截断、无退出码的输出证明通过。
- 不许信任代理自报成功；必须看 diff 并独立验证。
- 不许只验证一部分就声明全部完成。
- 没有证据时只能报告实际状态和缺口。
