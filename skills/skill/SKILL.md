# 编写技能

技能不是说明文，而是让模型在特定任务里稳定按步骤做事的规则。先证明没有技能时哪里会出错，再写最少的规则堵住出错的地方，最后验证它真的会被触发和遵守。

## 何时使用

**适合：**
- 某类任务会反复出现，且普通提示容易遗漏步骤
- 模型有稳定的错误倾向，或总是给自己找借口跳过某些步骤
- 需要按需加载的领域规则、流程或模板

**不适合：**
- 一次性项目观察
- 已经能用代码、lint、测试、hook 强制的机械规则
- 单个项目的长期背景说明（优先写 `CLAUDE.md` 或项目规则）

## 流程

### 写法要求

1. `name` 与目录名一致，只用字母、数字、连字符。
2. `description` 只写触发条件，不总结工作流。
3. 内容写动作链、停止条件、反模式和必要模板。
4. 长示例、矩阵、报告放 `references/`，且必须被 `SKILL.md` 明确说明何时读取。
5. 不要写空泛理念、项目游记、无触发条件的资料堆叠。

### Description 规则

`description` 决定模型是否加载技能，所以要写"什么时候用"，不要写"技能会做什么"。

```yaml
# 好：触发条件明确
description: 在遇到任何 bug、测试失败或意外行为时使用，先于提出修复方案

# 差：把流程写进 description，模型可能不再读取完整内容
description: 调试时先读错误、复现、找根因、写测试、修复并验证
```

### 最小流程

1. 写 2-3 个压力 prompt：正例、反例、容易跳过的逃逸例。
2. 不带技能跑一遍，记录模型哪里失败。
3. 只针对这些失败写或改 `SKILL.md`。
4. 带技能再跑同样 prompt，确认会触发、用户可见回复自然、会按内容行动。
5. 检查没有空引用、没有过期 skill 名、没有指向不存在的 reference。

### 质量检查

- 入口能被 `description` 正确触发。
- 内容能回答：先做什么、何时停止、下一步交给哪个 skill。
- 该读 reference 的地方写清楚，不需要的 reference 不保留。
- 规则不和 `using`、`think`、`plan`、`execute`、`verify` 冲突。
- 改完后运行插件校验。

<constraints>
- 禁止在 description 里总结工作流（只写触发条件）
- 禁止写空泛理念或无触发条件的资料堆叠
- 禁止保留没有被 SKILL.md 引用的 reference 文件
- 禁止写与 using/think/plan/execute/verify 冲突的规则
</constraints>

## 警示信号

| 念头 | 现实 |
|---|---|
| "多生成一点以后总会用到" | 无触发条件就是噪音 |
| "把扫描报告放进 reference" | reference 必须被某个 skill 读取 |
| "直接覆盖旧规则" | 先读旧内容，说明保留、追加或替换策略 |
| "只看 README 就够了" | 必须读真实配置和代表文件 |

## 集成

- 修改流程入口或触发规则前，先理解 `claude-workflow:using`。
- 修改实现前对齐流程时，先用 `claude-workflow:think`。
- 修改测试纪律时，先理解 `claude-workflow:test`。
- 宣称完成前，必须用 `claude-workflow:verify`。

## 沟通规范
- 用户看不到工具调用和思考过程，只看到你的文字输出
- 回复中不要出现本文件里的流程术语
- 用日常口语描述你在做什么："我先看看代码" / "写好了，测试通过" / "有个问题需要你确认"
- 匹配用户的说话风格——用户简短你就简短，用户详细你就详细
