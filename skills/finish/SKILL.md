# 安全收尾

finish 结束一段已经验证过的工作。它不负责证明完成——那属于 `verify`。

<HARD-GATE>
没有当前任务上下文中的新鲜验证证据时，禁止建议提交、推送、创建 PR 或合并。

禁止在用户明确确认前执行破坏性 git 操作。丢弃工作、删除分支、重置历史必须单独二次确认。
</HARD-GATE>

## 前置条件

必须满足至少一项：刚完成 `verify` 且通过；用户明确要求“提交 / 收尾 / 结束这项工作”；已有当前任务上下文中可引用的新鲜验证命令和输出。没有验证证据则先转 `verify`。

## 清单

1. **确认验证证据** — 列出刚运行过的命令和结果。
2. **查看工作区** — `git status --short`。
3. **概括变更** — 只总结本轮相关改动；不假装理解未读 diff。
4. **给出收尾选项** — 用户只说“收尾”时给选项；明确说“提交”时进入提交前确认。
5. **按选择执行** — 只执行用户选中的动作。
6. **执行后复核** — 提交后看 status；推送后报告结果；保留则说明当前状态。

## 收尾选项

```text
1. 提交当前工作
2. 推送 / 创建 PR
3. 保留当前工作，不提交
4. 丢弃当前工作
```

- 选项 1：提交前展示待提交文件摘要和建议 commit message。
- 选项 2：先确认已有提交；没有就先建议选项 1。
- 选项 3：不执行任何写操作，只说明工作区状态。
- 选项 4：二次确认并明确会丢弃哪些文件；未确认不得执行。
- 用户明确说“提交一下”时不重复四选一；直接展示验证证据、待提交文件摘要和建议 commit message，等确认后提交。

## 命令边界

允许：`git status --short`、`git diff --stat`、`git diff --cached --stat`、`git add <明确文件>`、`git commit -m "<message>"`、`git push` 或建 PR（先确认目标分支和远端）。

未经用户明确点名并确认，禁止直接执行：`git reset --hard`、`git checkout -- <path>`、`git restore <path>`、`git clean`、删除分支、force push。用户说“丢弃”仍要先二次确认具体范围。

## 提交前检查

有新鲜验证证据；已读 `git status --short`；待提交文件属于本轮工作；无计划外敏感 / 临时 / 大文件；commit message 简短准确。发现不属于本轮的改动，不自动纳入提交，列出让用户决定。

## 交付话术

```text
验证证据：
- `<命令>`：通过

当前工作区：
<git status --short 摘要>

收尾选项：
1. 提交当前工作
2. 推送 / 创建 PR
3. 保留当前工作，不提交
4. 丢弃当前工作

请选择一个选项。
```

## 警示信号

| 念头 | 现实 |
|---|---|
| "测试没跑但可以先提交" | 没有验证证据不能建议提交。 |
| "用户说收尾就是提交" | 收尾是选项，不是默认提交。 |
| "丢弃当前工作应该直接 reset" | 破坏性动作必须二次确认。 |
| "这些未跟踪文件应该一起提交" | 未跟踪文件必须逐项确认归属。 |
| "PR 可以顺手开" | 推送和 PR 都要确认远端、分支和目标。 |

## 何时不激活

- 还没验证完成，应该先用 `verify`。
- 还在执行计划，应该继续 `execute`。
- 用户要求代码评审，应该用 `review`。
- 用户要求隔离工作区，应该用 `worktree`。
