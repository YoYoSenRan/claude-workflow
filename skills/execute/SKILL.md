# 执行计划

读取已批准的计划，先做批判性复核，再按任务执行。计划是契约，不是建议。

<HARD-GATE>
遇到阻塞点必须停：计划含糊、文件不存在、命令无法运行、测试失败反复出现、实现需要超出计划范围。

禁止跳步骤、改计划、猜测意图、编造预期输出。
</HARD-GATE>

## 前置条件

必须满足：用户明确要求执行计划；存在书面计划文件，或用户贴出完整计划；计划任务有明确 涉及文件 / 步骤 / 运行 / 预期 / 验证 / 完成标准。没有计划则停止，转回 `plan`。

## 清单

1. **读取计划** — 一次性读完整计划。
2. **批判性复核** — 对照阻塞点表检查路径、命令、任务顺序、缺失步骤和含糊点。
3. **建立 TodoWrite** — 每个任务一项，不为每个步骤建任务。
4. **逐任务执行** — 只按计划列出的文件和步骤做。
5. **逐步验证** — 每个运行命令都真实执行并读取输出。
6. **遇阻即停** — 报告具体任务 / 步骤 / 输出，不猜。
7. **完成后转 verify** — 完成声明前进入 `verify` 或执行等价新鲜验证。

## 批判性复核

执行前对照下方阻塞点表逐项检查。任一项失败，停止并报告：

```text
计划阻塞：任务 N / 步骤 M 存在 <问题>。
原因：<为什么会导致执行失败或偏离需求>。
建议：回到 plan 修正后再执行。
```

转向规则：计划错但需求没变 → 回 `plan` 修订；需求或范围已变 → 回 `think` 重新确认；执行中发现未知 bug → 转 `debug`。

## 执行规则

- 每次只执行当前任务，完成并验证后才进入下一个。
- 不加计划外功能、不重构计划外代码、不修计划外 bug（只在 `> Note:` 提示）。
- 只更新计划里的 `状态` 和勾选框；不改写任务内容，除非用户明确要求回 `plan` 修订。
- 验证失败可围绕当前步骤做一次最小修正；仍失败就停。

## 阻塞点

| 阻塞点 | 处理 |
|---|---|
| 计划路径不存在 | 停，要求提供正确计划或回 `plan` |
| 文件路径不存在且未标新建 | 停，报告路径问题 |
| 步骤含糊 | 停，列出候选解读 |
| 运行命令不存在 | 停，报告命令问题 |
| 预期不具体 | 停，要求修计划 |
| 验证或完成标准不可验证 | 停，要求修计划 |
| 测试失败超过 1 次最小修正 | 停，报告输出 |
| 需要计划外文件 | 停，说明超出范围 |
| 发现 bug 根因不明 | 转 `debug` |
| 计划本身错误但需求没变 | 停，建议回 `plan` 修订原计划 |
| 需求已经变化 | 停，建议回 `think` 重新确认范围 |

## subagent 边界

`execute` 不承担完整 subagent-driven-development。原因：本插件未提供 实现者 / 规格评审 / 代码评审 等执行类子代理，没有其完整提示词，也没有自动化验证子代理交接行为。

任务过大不适合内联执行时，停止并建议：

```text
这个计划不适合内联执行。建议先补子代理工作流或把计划拆小。
```

不要假装已做过 subagent review。

## 完成条件

一个任务完成必须满足：计划列出的文件已按步骤处理；运行命令已执行且输出符合预期；没有引入计划外变更；TodoWrite 中标为 completed。

全部任务完成后不直接声称“完成”；先进入 `verify` 或运行计划指定的最终验证，有新鲜证据后再汇报。

## 警示信号

| 念头 | 现实 |
|---|---|
| "这一步大概是这个意思" | 大概就是阻塞点，停。 |
| "计划漏了，我顺手补上" | 顺手就是改计划，停。 |
| "测试失败像环境问题" | 没证据不能跳过。 |
| "先做后面的任务" | 顺序可能有依赖，不能跳。 |
| "这个适合派子代理" | 当前没有完整 subagent 工作流，先停或拆小。 |

## 何时不激活

- 没有书面计划，应该用 `plan`。
- 需求还没确认，应该用 `think`。
- 是 bug 根因排查，应该用 `debug`。
- 只是单文件小改，不需要执行计划。
