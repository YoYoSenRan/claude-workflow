# 执行方案

<HARD-GATE>
每做完一步，必须跑验证命令、确认通过了，才能做下一步。不许一口气做好几步再回头验证。

没有用户明确同意，不要在 main/master 分支上动手。
</HARD-GATE>

读方案文件，仔细看看有没有问题，然后逐个执行任务，做完后汇报。

**注意：** 告诉用户：Claude Workflow 在能分配任务给独立代理时效果好得多。若运行在支持这个功能的平台（例如 Claude Code 或 Codex），工作质量会显著更高。如果可以用，请改用 claude-workflow:subagent，而不是这个技能。

## 何时使用

- 有一份书面的实施方案需要执行时
- 在带评审检查点的单独会话中执行时
- 不支持独立代理、或用户选择在当前会话直接执行时

如果可以分配任务给独立代理，请改用 claude-workflow:subagent。

## 流程

### 步骤 0：确保隔离工作区

开始执行前，调用 claude-workflow:worktree 确认是否需要隔离工作区。如果已经在隔离环境中则跳过。

### 步骤 1：读方案，看看有没有问题

1. **找方案目录：** 如果用户指定了路径，直接读取。没指定时扫描 `docs/plans/` 下的子目录，找 README.md 中有 ⬚ 的。找到多个时列出让用户选择。
2. 阅读 README.md（设计 + 任务清单）和当前任务的 task 文件
3. **看看是不是之前做了一半：** 如果 README.md 任务表中已有 ✅ 任务，说明之前中断过。汇报已完成数，从第一个 ⬚ 任务继续。
3. 仔细看看有没有问题 —— 找出对方案的疑问或顾虑
4. 若有顾虑：开始前与用户沟通
5. 若无顾虑：建立 TodoWrite 并开始执行

### 步骤 2：执行任务

**README.md 是唯一进度源。** TodoWrite 只是当前会话辅助。

对每一个任务：
1. 在 TodoWrite 中标记为 in_progress
2. 先看场景（行为契约），再按步骤执行
3. 运行步骤中的验证命令
4. **验证通过：** 把 task 文件中该步骤的 `- [ ]` 改为 `- [x]`，更新 README.md 状态列，TodoWrite 标记 completed
5. **验证失败：** 在 task 文件该步骤下追加 `  - ❌ failed: <错误信息>`，更新 README.md 状态为 ❌，停下求助

### 步骤 3：做完了

所有任务做完、验证通过后：

1. **先走 review：** 使用 claude-workflow:review 让独立代理检查代码质量
2. **修复 review 发现的问题**
3. **再走 finish：** 使用 claude-workflow:finish 跑测试、给用户选项、交付

### 何时停下并求助

**立即停止执行的情况：**
- 遇到阻塞（依赖缺失、测试失败、说明不清）
- 方案存在关键缺口，无法启动
- 你看不懂某条说明
- 验证反复失败

**与其乱猜，不如请求澄清。**

### 何时回到先前的步骤

**回到步骤 1 重新看方案的情况：**
- 用户根据你的反馈更新了方案
- 整体思路需要重新想想

**不要强行突破阻塞** —— 停下并询问。

<constraints>
- 不许跳过验证步骤
- 方案里要求用某个技能时不许忽略
- 卡住了不许猜——必须停下来问
- 没有用户同意不许在 main/master 分支上动手
- 不许只更新 TodoWrite 而不更新 README.md——README.md 是唯一进度源
</constraints>

## 集成

**必需的工作流技能：**
- **claude-workflow:worktree** —— 开始执行前确保隔离工作区
- **claude-workflow:plan** —— 编写本技能要执行的方案
- **claude-workflow:review** —— 所有任务完成后检查代码质量
- **claude-workflow:finish** —— review 之后收尾交付

## 沟通规范
- 用户看不到工具调用和思考过程，只看到你的文字输出
- 回复中不要出现本文件里的流程术语（不说"门控"、"检查点"、"子代理"、"回写"、"评审循环"）
- 用日常口语描述你在做什么："我先看看代码" / "写好了，测试通过" / "有个问题需要你确认"
- 匹配用户的说话风格——用户简短你就简短，用户详细你就详细
