# 写实现计划

把已确认的需求拆成别人能照着执行的计划。计划是执行输入，不是思路草稿。

<HARD-GATE>
需求未确认时，禁止写计划；先回 `think`。

计划禁止占位符。每个实现步骤必须有精确文件路径、具体操作、验证命令和预期结果。缺任一项，就不是可执行计划。
</HARD-GATE>

## 前置条件

进入本 skill 前必须满足至少一项：

- 用户已经确认方案；
- 存在 `docs/specs/*.md`；
- 用户给出完整需求和边界；
- 用户明确要求“把这个方案写成计划”。

不满足时，停止并转回 `think` 对齐。

---

## 输出位置

默认写到：

```text
docs/plans/YYYY-MM-DD-<name>.md
```

`docs/roadmaps/` 只放项目路线图或阶段性改造方案；`docs/plans/` 放可执行任务计划。

如果目标仓库没有 `docs/plans/`，创建它。不要因为当前工作流仓库的目录结构，把实际项目里的任务计划写到 `docs/roadmaps/`。

---

## 清单

按顺序完成：

1. **确认输入** — 找到需求来源：用户确认文本、spec 文件或明确 prompt。
2. **范围检查** — 判断是否包含多个独立子系统；太大则拆多个计划。
3. **文档骨架** — 先写目标 / 范围 / 背景 / 涉及文件 / 验证方式 / 任务清单。
4. **文件结构** — 列出要创建或修改的文件，以及每个文件职责。
5. **任务拆分** — 每个任务应能独立理解、独立验证。
6. **步骤编写** — 每步写具体代码、命令或文件变更说明。
7. **测试策略** — 代码行为变化优先写失败测试；文档/配置变更说明为何不需要测试。
8. **占位符扫描** — 删除所有占位、模糊词和“参考上面”。
9. **自审** — 检查需求覆盖、路径、命令、类型一致性。
10. **交付用户** — 只交付计划，询问是否进入 `execute`。

---

## 范围检查

如果需求包含多个独立子系统，先拆计划，不要写一个巨型计划。

需要拆分的信号：

- 多个互不依赖的功能域；
- 前端、后端、数据库、部署同时大改；
- 单个计划无法独立验证；
- 任务超过 8 个且依赖关系复杂。

每个计划都应产出可运行、可验证的一部分软件。

---

## 计划文档格式

计划文件必须使用稳定结构，便于人类浏览，也便于 AI 断点续写：

````markdown
# 真实功能名 实现计划

**状态：** 草稿
**来源：** 用户确认文本、spec 路径或 issue 链接
**创建日期：** YYYY-MM-DD

## 目标

一句话说明最终要得到什么可验证结果。

## 范围

- 本计划包含的范围。
- 本计划不包含的范围。

## 背景

- 当前代码结构中的关键事实。
- 重要约束、兼容性要求或设计取舍。

## 涉及文件

- 新建：`path/to/new-file.ts` — 负责 ...
- 修改：`path/to/existing.ts` — 调整 ...
- 测试：`path/to/test.ts` — 覆盖 ...

## 验证方式

- `command` — 预期结果。

## 任务清单

### 任务 1：使用真实、短、动词开头的标题

**状态：** 待执行
**目的：** 本任务为什么存在，完成后解锁什么。

**涉及文件：**
- 新建：`path/to/file`
- 修改：`path/to/file:line`
- 测试：`path/to/test`

**步骤：**
- [ ] 步骤 1.1：具体动作。
  变更：写清要创建、替换或调整的内容；代码变更给完整片段。
  运行：`command`
  预期：具体通过、失败或输出条件。

**验证：**
- 运行：`command`
  预期：具体通过条件。

**完成标准：**
- 可观察的完成条件。
````

规则：

- `状态` 只能用 `草稿`、`已批准`、`待执行`、`执行中`、`已完成` 或 `已阻塞`，不要自造状态。
- `涉及文件` 是全局索引；每个任务的 `涉及文件` 是本任务触碰范围。
- `步骤` 是执行动作；`验证` 是任务级验证；`验证方式` 是计划级最终验证。
- `完成标准` 写人能验收的结果，不写主观描述。
- 执行阶段只允许更新 `状态` 和 checkbox；不得改写任务内容，除非用户明确要求回到 `plan` 修订。

---

## 文件结构

写任务前先列文件结构：

```markdown
## 文件结构

- 新建：`path/to/new-file.ts` — 负责 ...
- 修改：`path/to/existing.ts` — 调整 ...
- 测试：`path/to/test.ts` — 覆盖 ...
```

原则：

- 一个文件一个清晰职责。
- 遵循当前代码库已有风格。
- 不做与本需求无关的重组。
- 如果必须拆大文件，要说明拆分原因和影响面。

---

## 任务结构

每个任务使用固定结构：

````markdown
### 任务 N：真实任务名

**状态：** 待执行
**目的：** 本任务完成什么，以及为什么放在这个位置。

**涉及文件：**
- 新建：`path/to/file`
- 修改：`path/to/file:line`
- 测试：`path/to/test`

**步骤：**
- [ ] 步骤 1：<具体动作>
  ```ts
  // 完整代码；如果是文档/配置变更，写完整片段或明确替换内容
  ```
  运行：`<命令>`
  预期：`<预期输出或失败/通过条件>`

- [ ] 步骤 2：<具体动作>
  运行：`<命令>`
  预期：`<预期输出或通过条件>`

**验证：**
- 运行：`<命令>`
  预期：`<任务级通过条件>`

**完成标准：**
- <可观察完成条件>
````

规则：

- 路径必须精确。
- 命令必须可复制运行。
- 预期输出必须具体。
- 每个步骤只做一个动作，通常应能在 2-5 分钟内完成。
- 后续任务不能引用未定义的函数、类型或文件。
- 每个任务都必须能通过 `状态` 和 checkbox 表达进度。
- 不写 commit 步骤，除非用户明确要求计划中包含提交点。

---

## 测试策略

代码行为变化：

- 推荐 red -> green。
- 先写能失败的测试。
- 再写最小实现。
- 再跑相关测试确认通过。

文档、配置、纯说明类变更：

- 可以不写失败测试。
- 必须写清验证方式，例如格式检查、命令 dry-run、人工检查项。

不允许：

- 写“添加适当测试”。
- 写“覆盖边界情况”但不给具体测试。
- 把不适用的 TDD 模板硬塞进文档任务。

---

## 占位符黑名单

计划中禁止出现：

- `TBD`
- `TODO`
- `FIXME`
- `<placeholder>`
- `<name>`
- `类似任务 N`
- `参考上面`
- `添加适当处理`
- `完善逻辑`
- `处理边界情况`
- `实现该功能`

如果需要这些词出现在“反模式说明”里，必须明确它们是禁止项。

---

## 自审

交付前逐项检查：

- [ ] 是否覆盖全部需求。
- [ ] 是否明确不做什么。
- [ ] 是否列出所有要改的文件。
- [ ] 每个任务是否有 状态 / 目的 / 涉及文件 / 步骤 / 验证 / 完成标准。
- [ ] 每个命令是否有预期结果。
- [ ] 是否没有占位符。
- [ ] 任务顺序是否没有依赖倒挂。
- [ ] 测试策略是否和任务类型匹配。
- [ ] 输出路径是否为 `docs/plans/YYYY-MM-DD-<name>.md`。

如果任一项失败，先修计划，不交付。

---

## 可选评审

如果环境支持 subagent，且计划较大，可以使用：

```text
skills/plan/references/reviewer.md
```

但这是可选增强，不是硬门槛。没有 reviewer agent 时，不要假装已经评审。

---

## 交付话术

计划写完后：

```text
计划已写到 <path>。请先审一下范围和任务拆分。
如果方向没问题，下一步可以用 execute 按计划执行。
```

不要在 `plan` 内直接开始执行。

---

## 警示信号

| 念头 | 现实 |
|---|---|
| "先列大步骤就够了" | 执行者会卡住，必须写到文件和命令。 |
| "代码让执行阶段再想" | plan 的职责就是提前想清楚。 |
| "这个测试不适用但模板里有" | 删除不适用测试，换成真实验证。 |
| "写到 docs/roadmaps 也行" | 路线图和执行计划职责不同，任务计划必须进 `docs/plans/`。 |
| "评审子代理以后会有" | 没有真实 agent 时不能当成已评审。 |

## 何时不激活

- 需求还不清楚，应该先用 `think`。
- 用户要求直接执行已有计划，应该用 `execute`。
- 单文件小改，不需要计划。
- bug 根因不明，应该用 `debug`。
