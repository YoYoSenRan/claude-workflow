## 子代理辅助模式

被派遣起草或评审计划时，用本 skill。只产出草案、问题或建议；不替主智能体确认需求、批准计划、执行任务或修改计划状态。

# 写实现计划

把已确认的需求拆成别人能照着执行的计划。计划是执行输入，不是思路草稿。

<HARD-GATE>
需求未确认时，禁止写计划；先回 `think`。

计划禁止占位符。每个实现步骤必须有精确文件路径、具体操作、验证命令和预期结果。缺任一项，就不是可执行计划。

代码细节按风险分级：关键接口、类型签名、配置、迁移、对外行为必须写完整片段；普通内部实现写清输入、输出、调用点、约束和验证即可，不强迫在计划里提前写完整最终代码。
</HARD-GATE>

## 前置条件

进入前必须满足至少一项：用户已确认方案；存在 `docs/specs/*.md`；用户给出完整需求和边界；用户明确要求“把这个方案写成计划”。不满足则停止，转回 `think`。

不要为轻量小改写计划：单文件小改、文案 / 注释 / README 微调、明确局部替换、用户没要求拆任务且可一次完成。

应进入 `plan` 的信号：跨多文件或模块；多任务顺序有依赖；涉及公共 API、数据结构、迁移、部署；用户要求可恢复 / 可交接 / 可审查的计划；预计需要多轮执行。

## 输出位置

默认写到 `docs/plans/YYYY-MM-DD-<name>.md`。

`docs/roadmaps/` 只放路线图或阶段性改造方案；`docs/plans/` 放可执行任务计划。目标仓库没有 `docs/plans/` 就创建它。不要因为当前工作流仓库的目录结构，把任务计划写到 `docs/roadmaps/`。

## 范围检查

需求含多个独立子系统时先拆计划，不写巨型计划。拆分信号：多个互不依赖的功能域；前端、后端、数据库、部署同时大改；单计划无法独立验证；任务超过 8 个且依赖复杂。每个计划都应产出可运行、可验证的一部分软件。

## 计划文档格式

计划文件用稳定结构，便于人类浏览，也便于 AI 断点续写：

````markdown
# 真实功能名 实现计划

**状态：** 草稿
**来源：** 用户确认文本、规格说明路径或 issue 链接
**创建日期：** YYYY-MM-DD

## 目标

一句话说明最终要得到什么可验证结果。

## 范围

- 本计划包含的范围。
- 本计划不包含的范围。

## 背景

- 当前代码结构中的关键事实。
- 重要约束、兼容性要求或设计取舍。

## 涉及文件

- 新建：`path/to/new-file.ts` — 负责 ...
- 修改：`path/to/existing.ts` — 调整 ...
- 测试：`path/to/test.ts` — 覆盖 ...

## 验证方式

- `command` — 预期结果。

## 任务清单

### 任务 1：使用真实、短、动词开头的标题

**状态：** 待执行
**目的：** 本任务为什么存在，完成后解锁什么。

**涉及文件：**
- 新建：`path/to/file`
- 修改：`path/to/file:line`
- 测试：`path/to/test`

**步骤：**
- [ ] 步骤 1.1：具体动作。
  变更：写清要创建、替换或调整的内容；代码变更给完整片段。
  运行：`command`
  预期：具体通过、失败或输出条件。

**验证：**
- 运行：`command`
  预期：具体通过条件。

**完成标准：**
- 可观察的完成条件。
````

格式规则：

- `状态` 只能用 `草稿`、`已批准`、`待执行`、`执行中`、`已完成` 或 `已阻塞`，不自造。
- `涉及文件` 是全局索引；任务内 `涉及文件` 是本任务触碰范围。
- `步骤` 是执行动作；`验证` 是任务级验证；`验证方式` 是计划级最终验证。
- `完成标准` 写人能验收的结果，不写主观描述。
- 执行阶段只改 `状态` 和 勾选框；不改写任务内容，除非用户明确要求回 `plan` 修订。
- 计划要够执行，不是提前实现一遍代码；代码片段会快速过期时，写稳定接口、约束和验证条件。

## 文件与任务规则

文件结构：一个文件一个清晰职责；遵循已有代码风格；不做与本需求无关的重组；必须拆大文件时说明原因和影响面。

任务规则：

- 路径精确、命令可复制运行、预期输出具体。
- 每个步骤只做一个动作，通常 2-5 分钟可完成。
- 公开 API、类型、配置、迁移、命令参数必须给完整签名或片段；普通内部实现写明数据流、调用点、错误处理边界和可验证结果。
- 后续任务不能引用未定义的函数、类型或文件。
- 每个任务用 `状态` 和 勾选框 表达进度。
- 不写 commit 步骤，除非用户明确要求计划中包含提交点。

## 测试策略

代码行为变化：推荐 red → green（先写能失败的测试 → 最小实现 → 跑相关测试确认通过）。

文档、配置、纯说明类变更：可不写失败测试，但必须写清验证方式，例如格式检查、命令 dry-run、人工检查项。

不允许：写“添加适当测试”；写“覆盖边界情况”但不给具体测试；把不适用的 TDD 模板硬塞进文档任务。

## 占位符黑名单

计划禁止出现：`TBD`、`TODO`、`FIXME`、`<placeholder>`、`<name>`、`类似任务 N`、`参考上面`、`添加适当处理`、`完善逻辑`、`处理边界情况`、`实现该功能`。如果用于“反模式说明”，必须标明它们是禁止项。

## 自审

交付前逐项检查：是否覆盖全部需求；是否明确不做什么；是否列出所有要改的文件；每个任务是否有 状态 / 目的 / 涉及文件 / 步骤 / 验证 / 完成标准；每个命令是否有预期结果；是否没有占位符；任务顺序是否没有依赖倒挂；测试策略是否和任务类型匹配；输出路径是否为 `docs/plans/YYYY-MM-DD-<name>.md`。任一失败先修计划，不交付。

## 可选评审

环境支持 subagent 且计划较大时，可用 `references/reviewer.md` 做评审。这是可选增强，不是硬门槛。没有 reviewer agent 时，不要假装已评审。

## 交付话术

声明用 A 风格：进行打头，完成收口。

```text
> **🔧 plan** — <目的>

计划已写到 <path>。请先审一下范围和任务拆分。

> **✅ 完成** — 计划已落 `<path>` · 下一步 execute
```

不要在 `plan` 内直接开始执行。

## 警示信号

| 念头 | 现实 |
|---|---|
| "先列大步骤就够了" | 执行者会卡住，必须写到文件和命令。 |
| "代码让执行阶段再想" | plan 的职责就是提前想清楚。 |
| "这个测试不适用但模板里有" | 删除不适用测试，换成真实验证。 |
| "写到 docs/roadmaps 也行" | 路线图和执行计划职责不同，任务计划必须进 `docs/plans/`。 |
| "评审子代理以后会有" | 没有真实 agent 时不能当成已评审。 |

## 何时不激活

- 需求还不清楚，应该先用 `think`。
- 用户要求直接执行已有计划，应该用 `execute`。
- 单文件小改，不需要计划。
- bug 根因不明，应该用 `debug`。
