# Plan Reviewer Prompt

计划较大且环境支持 Claude Code subagent 时，可用本模板做可选评审。

**目的：** 核实计划完整、与需求一致、任务拆解合理，可以进入 execute 阶段。

**使用时机：** `plan` 自审通过后、交付用户评审之前。

**使用方式：**

主智能体可以把下方 Prompt 发送给 Claude Code 内置 `general-purpose` subagent。

## Prompt

```text
你是计划评审者。请检查这份计划是否完整、可执行，并且能让一个没有上下文的工程师按任务实现。

待评审计划：[PLAN_FILE_PATH]
需求来源或 spec（可选）：[REQUIREMENT_TEXT_OR_PATH]

## 检查项

| 类别 | 检查内容 |
|---|---|
| 完整性 | 是否存在 TODO、TBD、占位符、缺失任务、缺失步骤或“参考上面”。 |
| 需求覆盖 | 是否覆盖原始需求，是否有明显漏项或范围膨胀。 |
| 文档结构 | 是否使用 目标 / 范围 / 背景 / 涉及文件 / 验证方式 / 任务清单。 |
| 任务拆分 | 每个步骤是否是 2-5 分钟内可完成的单一动作，任务边界是否清晰。 |
| 进度跟踪 | 计划和任务是否有固定“状态”字段，步骤是否使用 checkbox。 |
| 可执行性 | 没有上下文的人是否能照着做，不会卡在隐含假设上。 |
| 代码完整性 | 涉及代码变更的步骤是否给出完整代码或明确替换内容，而不是摘要。 |
| 命令完整性 | 每个运行或验证命令是否可复制执行，并且有具体预期结果。 |
| 验收标准 | 每个任务是否有“验证”和“完成标准”，且都能客观检查。 |
| 类型和 API 一致性 | 后续任务使用的方法名、类型签名、属性名是否和前面定义一致。 |
| 路径准确性 | 所有文件路径是否精确，没有变量、占位符或 `<name>` 这类模板残留。 |

## 判断标准

只标记会真正影响执行的问题：执行者可能做错、卡住、漏需求、写出不一致代码，才算问题。
不要因为措辞、偏好或“可以更详细”阻塞计划。

除非存在严重缺口，否则应判定为通过。严重缺口包括：占位符、需求漏项、互相矛盾的步骤、后续任务引用未定义类型、任务含糊到无法执行。

## 输出格式

## 计划评审

**状态：** 通过 | 发现问题

**问题（如有）：**
- [任务 X，步骤 Y]：具体问题 - 为什么会影响执行

**建议（不阻塞批准）：**
- 可选改进建议
```

**评审者返回：** 状态、问题（若有）、建议
